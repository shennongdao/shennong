神农书库（ShenNong Library），一个去中心化且开源的小说评价平台。

一切知识产权和品牌所有权属于DAO(去中心化自治组织)，DAO由代币持有者进行治理，避免被单一实体控制。代币可通过发布评分、打标签、代码开发、完成推广任务等获得。

目标

- 方便读者找书
- 永久可用
- 抗审查
- 低成本

为什么要去中心化与开源？
对于一个收入可能为零的小众项目来说，依靠一个中心化的机构或个人长期维护不太现实。去中心化与开源能吸引更多开发者与内容贡献者，延长项目寿命：

- 上线后，神农小说可能长期都是一个小众平台，用户使用此平台都会担忧平台的寿命、平台死亡后个人数据(评分、评论、书架、书单)丢失、贡献记录丢失。去中心化与开源能保障fork项目能保留用户的个人数据与贡献记录，使所有人的利益在新fork项目上能继续维持。新项目如果不延续项目用户的贡献，将很难吸引本项目的用户迁移过去。
- 质押代币，更能保证评分、评论的真实性与透明性，而代币的发行如果依赖中心化机构，无法让人相信不会滥发。
- 优书网挂掉后，几个优书网替代品大都也停止维护了。所有人都能参与治理，早期参与者离开后，新参与者能继续通过代币参与治理、维护项目，最大程度延长项目寿命。

如何做到去中心化？

**域名**作为用户入口，是一个项目最重要的资产之一。DAO无法拥有传统域名的控制权，如果用传统域名，这一资产将不可避免地掌握在个人或中心化机构手中，DAO失去这一重要资产的风险太大，以前出现过这种情况。神农书库使用去中心化域名（ARNS）作为入口，使得域名的控制权掌握在DAO手中，避免域名变动造成用户流失。

**前端界面**部署在Arweave上永久存储，永远可访问。

**后端协议**部署在AO网络上，所有数据都会永久存储在Arweave，公开透明，当AO节点故障时，任何人都能通过这些数据恢复神农书库的运行，这也能避免项目失败后用户找不回自己的数据。

**网关**索引数据，提升用户访问速度。由于所有数据都在Arweave，任何人都能运行网关，当官方网关停止时，用户可在前端界面配置使用使用第三方网关。

用户如何在神农书库找书

- 标签筛选。一本书值不值得看，无非就是优缺点的权衡，优点（爽点）是不是足够吸引你读下去，缺点（毒点）是不是足够劝退你，两者一对比，心里就有数了。但各小说平台的标签不太准确，且没有全面的毒点标签，神农书库通过代币质押机制鼓励所有用户为小说打上正确全面的标签。
- 书库中按小说的评分排序
- AI找书：根据提示词找符合的标签

以上方法找书时可筛选指定小说范围：

- 限定男频、女频
- 最近一次更新时间
- 字数
- 剔除书架中的所有书
- 剔除黑名单中的所有书。不想看的书反复出现，特别烦人，可以设置成看过简介的书自动加入黑名单，也可以设置程浏览过的书自动加入黑名单，加入后可以按时间范围从黑名单中删除。

# 代币经济

代币中文名：神农书库代币
代币英文名：Shennong Library Token
代币符号：`SNLT`，测试币符号为`tSNLT`
代币精度：12

**价值来源**

- 每一个代币都对应着一分钟的劳动价值。需要换算成货币时，1个代币=中国国家统计局公布的最近一年城镇私营单位就业人员年平均工资/250/8/60，比如2026年6月一个代币=71590/250/8/60=0.59658 元
- 手续费。处罚、打赏、资助推广金时收取手续费
- 购买SNID用户名
- 流通价值。作者需要代币存入推广金，读者需要代币质押、打赏等
- 参与治理，决定项目发展方向
- 高级操作，比如计算量大的查询，有持币要求。
- 广告。这个很影响用户体验，作为长期不能达到收支平衡时才会采用得最后手段。

**代币发行原则**

- 零预挖、公平发行
- 代币历史累计最大发行量为 `100,000,000` 枚，此数量固定不可修改。
- 创世发行量为 `0`。
- 不向创始人、开发团队、投资人、顾问或社区进行任何预分配。DAO向开发者、推广者和内容贡献者支付后，代币才逐渐进入个人持有和市场流通。
- 所有代币只能由统一发行合约按日增发产生。
- 销毁的代币不恢复增发额度。
- 历史累计增发量达到 `100,000,000` 枚后，永久停止增发。

**每日增发上限**
每日增发上限应保证：即使每天都按照上限增发，也不能在协议启动后的十周年之前完成全部发行。
设：

- `MAX_SUPPLY`：历史累计最大发行量；
- `D10`：从协议启动日至十周年之间的每日增发周期数；
- `U`：每日增发上限。

计算方式：
`U = floor(MAX_SUPPLY ÷ D10)`

若协议启动日至十周年共有 `3653` 个每日增发周期，则：
`U = floor(100,000,000 ÷ 3653) = 27,374 枚/日`

因为：
`27,374 × 3653 = 99,997,222`
所以在第 `3653` 个每日增发周期结束后，累计发行量仍未达到一亿枚，至少还需要下一个每日周期才能完成发行。
协议正式启动日期确定后，应按照启动日至十周年的实际天数确定 `D10`，并在发行合约部署时固定每日增发上限。

**每日增发下限**
每日最低增发数量固定为：
`L = 2,737 枚/日`

每日增发区间为：
`2,737 ≤ E(t) ≤ 27,374`

即使前一日没有协议收入，次日仍然增发 `2,737` 枚代币，用于支持项目早期的开发、推广和内容建设。
若长期始终按照最低数量增发：

- 每年约增发 `999,005` 枚；
- 完成一亿枚发行约需 `100.1` 年。
  若长期始终按照最高数量增发，完成全部发行仍需超过十年。

**协议收入**
协议所有收入均使用本项目代币支付，不考虑其他代币、稳定币或法币收入。

设：

- `R(t-1)`：前一日最终确认的有效协议净收入；
- `E(t)`：根据收入计算出的当日理论增发数量；
- `Mint(t)`：当日实际增发数量。

有效协议净收入包括：

- 用户实际支付并最终归协议所有的手续费；
- 用户购买协议服务支付的代币；
- 挑战、仲裁等流程中最终归协议所有的费用；
- 其他真实用户支付并最终归协议所有的代币收入。

以下不计入有效协议收入：

- 每日增发产生的代币；
- 各级DAO之间的内部转账；
- DAO向任务执行者支付的报酬；
- 国库之间的资金调拨；
- 尚未最终处理的押金和保证金；
- 后续需要退还的款项；
- 用户退款和返佣；
- 发行合约、DAO国库、项目托管合约等协议控制地址支付的费用；
- 关联地址循环交易产生的虚假收入；
- 以提高次日增发量为目的制造的无真实经济意义收入。

前一日收入应当在扣除退款、返佣以及其他不归协议所有的部分后，再用于计算次日增发数量。

**每日增发公式**

每日理论增发数量为：
`E(t) = min(U, max(L, R(t-1)))`

代入初始参数：
`E(t) = min(27,374, max(2,737, R(t-1)))`

具体规则如下：

- 当前一日有效协议收入低于 `2,737` 枚时，次日增发 `2,737` 枚；
- 当前一日有效协议收入处于 `2,737` 至 `27,374` 枚之间时，次日增发数量等于前一日有效协议收入；
- 当前一日有效协议收入超过 `27,374` 枚时，次日只增发 `27,374` 枚。

示例：


| 前一日有效协议收入 | 次日理论增发数量 |
| -------------------: | -----------------: |
|                  0 |            2,737 |
|              1,000 |            2,737 |
|              2,737 |            2,737 |
|              5,000 |            5,000 |
|             15,000 |           15,000 |
|             27,374 |           27,374 |
|             50,000 |           27,374 |

超过每日增发上限的协议收入：

- 不累计为以后日期的增发额度；
- 不结转到下一日；
- 不进行补发；
- 仍然作为协议已经取得的收入，由协议国库按照治理规则使用。

**剩余发行量限制**

设：

- `C(t-1)`：截至前一日的历史累计增发量；
- `Remaining(t)`：当日开始前的剩余可发行数量。

计算方式：
`Remaining(t) = MAX_SUPPLY - C(t-1)`

当日实际增发数量为：
`Mint(t) = min(E(t), Remaining(t))`

完整公式为：
`Mint(t) = min(100,000,000 - C(t-1), 27,374, max(2,737, R(t-1)))`

当剩余可发行数量低于每日最低增发数量时，只增发剩余数量。

例如仅剩 `1,500` 枚尚未发行时：
`Mint(t) = 1,500`

累计增发量达到 `100,000,000` 枚后：
`Mint(t) = 0`

此后即使协议继续产生收入，也不再增发新代币。

**协议收入与新增代币的区别**

协议收入是已经发行并进入流通的代币重新回到协议：
`用户支付费用 → 协议收入国库`

每日增发是此前从未发行的新代币：
`发行合约铸造新代币 → 各级DAO资金池`

前一日收到多少协议收入，并不意味着这些收入代币被销毁。协议收入和次日新增代币是两笔独立资产：

- 收入代币进入协议收入国库；
- 新增代币按照资金分配规则进入各级DAO；
- 收入代币是否支出、保留或者销毁，由另外的国库治理规则决定；
- 即使收入代币被销毁，也不会恢复一亿枚的历史累计增发额度。

**每日结算流程**

1. 每日结算前一日已经最终确认的协议收入。
2. 扣除退款、返佣以及其他不属于协议的金额。
3. 排除内部转账、协议控制地址付款和循环交易等无效收入。
4. 得到前一日有效协议净收入 `R(t-1)`。
5. 按照以下公式计算理论增发数量：
   `E(t) = min(27,374, max(2,737, R(t-1)))`
6. 计算剩余可发行数量：
   `Remaining(t) = 100,000,000 - C(t-1)`
7. 计算当日实际增发数量：
   `Mint(t) = min(E(t), Remaining(t))`
8. 将当日新增代币按照当日生效的资金分配票和专业委托票，递归分配到各级DAO资金池。
9. 更新历史累计增发量：
   `C(t) = C(t-1) + Mint(t)`

**参数汇总**

- 历史累计最大发行量：`100,000,000` 枚；
- 创世发行量：`0`；
- 每日最低增发数量：`2,737` 枚；
- 每日最高增发数量：`27,374` 枚；
- 增发依据：前一日有效协议净收入；
- 收入单位：本项目代币；
- 收入低于下限：按照下限增发；
- 收入处于上下限之间：按照实际收入数量增发；
- 收入超过上限：按照上限增发；
- 超额收入不结转、不补发；
- 销毁代币不恢复发行额度；
- 累计增发达到一亿枚后永久停止增发。

# DID

**定位**

- 平台使用自定义去中心化身份，不要求符合 W3C DID 标准。
- 身份称为 `SNID（Shennong Identity）`，产品层可以继续称为 DID。
- SNID 是平台内永久不变的身份主键。
- 评论、评分、标签、投票、声誉、处罚记录等全部绑定 SNID。
- 钱包只是 SNID 当前的控制者，可以更换。
- 平台代币、DEX 仓位、保证金和奖励归属于钱包地址，不归属于 SNID。

**SNID 格式**

完整格式：

`snid:<16位Base64URL字符>`

示例：

`snid:K7mP9x2D8r5Qv3Ca`

字符范围：

`A-Z`、`a-z`、`0-9`、`-`、`_`

校验规则：

`^snid:[A-Za-z0-9_-]{16}$`

规则：

- 标识部分固定为 16 个字符。
- 使用 Base64URL 编码。
- 区分大小写。
- 不允许任何空白字符。
- 不允许用户自行选择。
- 创建后永久不变。
- 钱包迁移后 SNID 不变。
- Registry 内部可以只保存 16 位裸标识，对外显示时再添加 `snid:` 前缀。

16 个 Base64URL 字符提供的信息量为：

$16 \times 6 = 96\ \text{bit}$

SNID 只用于唯一标识，不是私钥，因此 96 bit 足够。Registry 创建 SNID 时仍然必须检查是否已经存在，排除理论上的碰撞。

推荐生成方式：

`SNID = Base64URL(SHA-256("shennong-snid-v1" || RegistryProcessId || CreateSNIDMessageId) 的前 12 字节)`

12 字节等于 96 bit，Base64URL 编码后正好得到 16 个字符，不需要 `=` 填充。

**AO 架构**

不为每个 SNID 单独创建 AO Process。

使用一个共享的 `DIDRegistry Process` 管理全部身份。

状态示例：

```
DIDs = {
        [snid] = {
            owner = "wallet-address",
            username = nil,
            displayName = nil,
            profileTxId = nil,
            controllerEpoch = 1,
            delegates = {},
            status = "active",
            createdAt = 0,
            lastTransferredAt = nil
        }
    }
```

```
UsernameToSNID = {
        -- ["async_nomad"] = "K7mP9x2D8r5Qv3Ca"
    }
```

```
WalletToSNIDs = {
        -- ["wallet-address"] = {
        --     ["K7mP9x2D8r5Qv3Ca"] = true
        -- }
    }
```

SNID 在语义上是共享 Registry 中唯一、不可拆分、可转移的身份资产，类似 NFT，但不是“一个身份对应一个 AO Process”的 Atomic Asset。

Registry 只处理低频身份操作：

- 创建 SNID；
- 查询当前控制者；
- 购买用户名；
- 设置公开资料；
- 添加或撤销 Delegate；
- 发起身份迁移；
- 接受身份迁移；
- 取消身份迁移；
- 停用身份；
- 更新 `controllerEpoch`。

评论、评分、标签、投票等高频操作由独立业务 Process 处理，不能全部写入 DID Registry。

**身份与钱包的关系**

每个 SNID 有一个当前 Owner：

```
DIDs[snid].owner = walletAddress
```

Owner 可以：

- 转移 SNID；
- 设置和撤销 Delegate；
- 修改身份资料；
- 设置恢复方式；
- 停用 SNID。

钱包拥有：

- 平台代币；
- AR、AO 等资产；
- DEX 仓位；
- 已获得奖励；
- 未结算保证金；
- 操作产生的经济责任。

SNID 拥有：

- 身份编号；
- 用户名；
- displayName；
- 个人资料入口；
- 评论历史；
- 评分历史；
- 标签贡献历史；
- 投票历史；
- 声誉；
- 处罚记录；
- 控制权迁移历史。

代币余额应记录在钱包地址下：

```
Balances[walletAddress] = tokenAmount
```

不应记录为：

```
Balances[snid] = tokenAmount
```

这样可以直接兼容 AO Token、DEX、质押和其他 DeFi 协议。

**Delegate**

Owner 可以授权临时钱包代表 SNID 执行日常操作。

状态示例：

```
delegates = {
        ["temporary-wallet-address"] = {
            scopes = {
                comment = true,
                rating = true,
                tag = true
            },
            expiresAt = 1785000000
        }
    }
```

Delegate 可以被限制为：

- 只能评论；
- 只能评分；
- 只能打标签；
- 只能执行普通社区操作；
- 只能在指定时间内有效。

Delegate 不能：

- 转移 SNID；
- 修改恢复方式；
- 更换 Owner；
- 停用 SNID；
- 修改其他 Delegate。

推荐使用方式：

```
硬件钱包：Owner
    浏览器临时钱包：Delegate
```

这样用户不需要每次评论、评分或打标签都连接硬件钱包。

**操作绑定**

评论、评分、标签、投票等操作必须同时记录 SNID 和实际签名钱包。

示例：

```
{
      "action": "CreateTag",
      "snid": "K7mP9x2D8r5Qv3Ca",
      "controllerEpoch": 3,
      "signerWallet": "wallet-address",
      "fundingWallet": "wallet-address",
      "rewardWallet": "wallet-address",
      "bookId": "book-id",
      "contentTxId": "arweave-tx-id",
      "nonce": 183,
      "bond": "1000000",
      "timestamp": 1782540000
    }
```

业务 Process 验证：

1. SNID 是否存在；
2. SNID 状态是否为 `active`；
3. 签名钱包是否为当前 Owner；
4. 或签名钱包是否具有对应操作的 Delegate 权限；
5. Delegate 是否已经过期；
6. `controllerEpoch` 是否与 Registry 当前值一致；
7. Nonce 是否有效，防止消息重放；
8. 是否已经支付费用或锁定保证金；
9. 经济权利与责任归属于哪个钱包。

历史内容归属于 SNID，但经济权利与责任归属于实际钱包。

**身份迁移**

SNID 可以从临时钱包迁移到硬件钱包。

迁移后：

- SNID 不变；
- 用户名不变；
- displayName 不变；
- 评论、评分、标签、投票和声誉不变；
- Owner 更新为新钱包；
- `controllerEpoch` 加 1；
- 原有 Delegate 默认全部撤销；
- 旧钱包中的代币、奖励和 DEX 仓位不会转移；
- 旧钱包承担的未结算保证金和处罚责任不会转移。

不使用普通 NFT 的即时转账接口，使用双向确认：

- `ProposeTransfer(NewOwner)`
- `AcceptTransfer(SNID)`
- `CancelTransfer()`
- `FinalizeTransfer()`

推荐流程：

```
旧 Owner 发起迁移
            ↓
    新 Owner 签名接受
            ↓
    进入等待期
            ↓
    完成迁移
            ↓
    controllerEpoch 加 1
```

存在以下情况时，可以禁止迁移：

- SNID 正在被挑战；
- 存在未完成仲裁；
- 存在待执行的身份处罚；
- SNID 已被冻结；
- 存在尚未结束的高影响治理操作。

链上无法区分正常的钱包迁移和身份出售，因此只能通过等待期、公开迁移记录和治理冷却期提高身份交易成本。

SNID 不提供彻底删除历史的 `Burn` 操作，只允许：

`DeactivateSNID`

停用后：

- 历史仍可查询；
- 用户名继续被占用；
- 不能再评论、评分、打标签或投票；
- 不能通过销毁身份逃避负面记录。

**创建 SNID**

创建 SNID 对用户免费。

可以由官方或第三方 Relayer 代付相关网络成本：

```
用户签署 CreateSNID 请求
            ↓
    Relayer 提交消息并支付网络成本
            ↓
    DIDRegistry 创建 SNID
            ↓
    SNID 归属于用户钱包
```

协议保持无许可：

- 用户可以使用平台 Relayer 免费创建；
- 用户也可以自行支付网络成本创建；
- 第三方可以运行自己的 Relayer；
- Relayer 的补贴限制不影响协议本身的访问权。

免费 SNID 不自动获得任何影响力：

- 治理权重为 0；
- 推荐排序权重为 0；
- 标签确认权重为 0；
- 挑战权重为 0；
- 不自动获得奖励资格。

普通评论不要求持有代币、支付费用或质押保证金，也不自动获得任何系统影响力。

评分、打标签、投票、挑战、治理，以及要求改变公共展示状态或处罚用户的正式评论举报等操作，必须根据其影响范围承担一种或多种成本：

- 不可退还费用；
- 可退还保证金；
- 可罚没质押；
- 锁定代币；
- 已积累声誉；
- 历史准确率。

**用户名**

用户名采用一次性购买，不使用质押、续费、租赁、释放或回收机制。

用户名规则：

- 全局唯一；
- 一个 SNID 最多购买一个用户名；
- 用户名购买后不能修改；
- 用户名永久绑定 SNID；
- 用户名不能单独转让；
- SNID 转移时用户名随 SNID 一起迁移；
- SNID 停用后用户名永久冻结，不重新出售；
- 用户名购买费用不可退还；
- 购买费用进入协议公共资金池。

推荐格式：

`^[a-z][a-z0-9_]{2,23}$`

即：

- 长度为 3～24 个字符；
- 必须以小写英文字母开头；
- 只能使用小写英文字母、数字和下划线；
- 不允许任何空白字符；
- 不允许大写字母；
- 不允许中文；
- 不允许 Emoji；
- 不允许零宽字符和控制字符。

有效示例：

- `async_nomad`
- `herbert`
- `book123`

无效示例：

- `AsyncNomad`
- `async nomad`
- `异步客`
- `_async`
- `async-nomad`

系统保留名称包括但不限于：

- `admin`
- `administrator`
- `system`
- `official`
- `support`
- `moderator`
- `dao`
- `root`
- `null`
- `undefined`

用户名定价可以只按长度分级：

- 3 个字符：高价；
- 4 个字符：中价；
- 5～24 个字符：统一普通价格。

对应治理参数：

- `UsernamePrice3`
- `UsernamePrice4`
- `UsernamePriceNormal`

购买接口：

`BuyUsername(SNID, Username)`

Registry 验证：

1. 调用钱包是否为 SNID 当前 Owner；
2. SNID 是否尚未购买用户名；
3. 用户名格式是否合法；
4. 用户名是否未被占用；
5. 用户名是否不属于保留词；
6. 支付金额是否足够；
7. 写入 `UsernameToSNID`；
8. 写入 `DIDs[snid].username`。

两个用户同时购买同一个用户名时，以 DIDRegistry 中先成功执行的消息为准。

**displayName**

只有购买用户名后，才允许设置 `displayName`。

未购买用户名时：

- 使用系统生成名称；
- 不能自行设置 displayName；
- 仍然可以正常评论、评分、打标签、投票和积累声誉。

例如：

`用户 K7mP9x`

购买用户名后：

```
displayName：异步客
    username：@async_nomad
```

公开展示时应始终同时显示 displayName 和唯一用户名：

```
异步客
    @async_nomad
```

不能只显示 displayName，因为 displayName 可以重复。

displayName 规则：

- 可以重复；
- 可以修改；
- 可以使用中文；
- 可以使用普通空格；
- 可以使用 Emoji；
- 不能是纯空白；
- 禁止换行和制表符；
- 禁止控制字符和零宽字符；
- 去除首尾空白；
- 设置长度上限，例如 24 个 Unicode 字符；
- 禁止使用系统保留名称；
- 禁止伪造认证标记。

购买用户名不代表身份已经认证。

例如：

```
displayName：刘慈欣
    username：@liucixin
```

不能据此认定该用户是真实作者。作者、平台、DAO 和机构需要单独的认证机制。

**公开展示**

未购买用户名时：

`用户 K7mP9x`

身份详情中显示：

`SNID：snid:K7mP9x2D8r5Qv3Ca`

购买用户名后：

```
异步客
    @async_nomad
```

SNID 不需要在普通评论区持续显示，但在身份详情、复制、验证和区块浏览器中必须可以查看。

**最终关系**

SNID：

- 免费创建；
- 自动生成；
- 永久不变；
- 不能由用户选择；
- 可以在钱包之间迁移；
- 绑定评论、评分、标签、投票、声誉和处罚历史。

用户名：

- 一次性购买；
- 全局唯一；
- 不能修改；
- 永久绑定 SNID；
- 不能单独转让；
- 随 SNID 一起迁移。

displayName：

- 购买用户名后才能设置；
- 可以重复；
- 可以修改；
- 用于公开展示；
- 必须和唯一用户名一起显示。

钱包：

- 控制 SNID；
- 持有代币；
- 与 DEX 集成；
- 支付费用和保证金；
- 接收奖励；
- 承担经济处罚。

# 声誉

**声誉生成**

- 每个DAO分别记录独立声誉。
- 完成开发DAO任务，只获得开发DAO声誉。
- 完成推广DAO任务，只获得推广DAO声誉。
- 完成内容DAO任务，只获得内容DAO声誉。
- 下级DAO任务产生的声誉可以同时计入该下级DAO及其直接上级DAO。
- 上级DAO声誉不能反向生成具体下级DAO声誉。

任务通过最终验收后：
`1分钟已经确认的人类劳动时间 = 获得对应DAO的1点声誉`

声誉具有以下性质：

- 不可转让；
- 不可出售；
- 不可委托；
- 不随代币转账而转移；
- 获得代币后即使出售，已经取得的声誉仍然保留；
- 同一份人类劳动贡献只能生成一次声誉；
- 问题任务被撤销时，对应声誉可以撤销。

任务付款必须区分：

`任务总付款 = 劳动报酬 + 成本报销 + 采购费用 + 保证金返还 + 其他款项`

只有已经确认的人类劳动时间产生声誉。

代币也以人类劳动时间作为价值参考，`1个代币`对应`1分钟人类劳动时间`。但代币可以按照任务规则换算并用于补贴服务器、广告、工具、代理、第三方采购等其他货币开销。

声誉不用于补偿其他货币开销，只记录人类劳动贡献。

以下款项不产生声誉：

- 服务器、广告、工具等费用报销；
- 第三方采购费用；
- 押金或保证金返还；
- 投资款；
- 流动性奖励；
- 赔偿；
- DAO之间的资金拨款；
- 与实际劳动无关的其他奖励。

例如某推广任务总付款为 `5,600代币`：

- 劳动报酬：`600代币`
- 广告费用报销：`5,000代币`
- 确认人类劳动时间：`600分钟`
- 产生声誉：`600点`

**累计声誉、活跃声誉与专业票权**
声誉分为：

- 累计声誉：记录历史有效贡献，用于判断正式成员资格；
- 活跃声誉：反映近期专业贡献，用于限制当前可承载的委托票。

活跃声誉可以采用365天线性衰减：
`某项声誉当前有效值 = 原始声誉 × max(0, 1 - 获得后经过天数 ÷ 365)`

例如：

- 刚获得的 `100点`声誉，活跃值为 `100点`；
- 半年后约为 `50点`；
- 超过365天后不再提供专业票权；
- 该声誉仍然保留在累计声誉记录中。

**声誉防刷**
任务只有同时满足以下条件才生成声誉：

- 明确区分劳动报酬与成本报销；
- 有明确交付物、预算和验收条件；
- 完成后经过无利益冲突的验收或挑战期；
- 劳动报酬已经实际支付；
- 任务已经通过规定的挑战期；
- 相关利益关系已经公开。

还应遵守：

- 一个正常任务不能通过人为拆分增加总声誉；
- 同一成果不能通过多个地址重复获取声誉；
- 自己不能单独审批、验收和支付自己的任务；
- 关联方任务必须披露，并由无关联人员验收；
- 重复、抄袭、自动生成且没有实际价值的成果不产生声誉；
- 使用AI提高效率不自动减少报酬，但任务预算应参考公开竞标、实际交付价值和合理工作量；
- 任务被认定无效、虚假或抄袭时，撤销该任务产生的声誉；
- 审批者或验收者参与造假时，也应承担声誉扣除和资格处罚。

- Watchlist：观察名单。
- Quarantined：隔离状态。
- Blocked：暂停登记。

# 作者

作者信息：

- 作者id：
  平台信息：{小说平台id: {作者在该平台的id, 作者名}}
  主小说平台，该作者的所有小说要在信息中另外提示作者在主小说平台的名字，点击到作者详情要显示其在所有小说平台的名字。

作者状态：

- Risk-2：多本小说确认刷量或刷量比例较高。

author_key = hash(platform_id + author_id)

# 小说书籍信息

每本小说记录以下信息：

- `bookID`：本协议内部的小说唯一标识；
- `firstPublishPlatformID`：首发平台；
- `firstPublishPlatformBookID`：小说在首发平台的书籍ID；
- `title`：书名；
- `authorName`：作者名；
- `authorID`：作者在首发平台的ID；
- `readURLs`：全部已知阅读链接；
- `coverURL`：封面链接；
- `description`：简介；
- `channel`：男频、女频或其他频道；
- `wordCount`：当前字数；
- `latestChapter`：最新章节；
- `sourceUpdatedAt`：来源平台显示的最近更新时间；
- `completionStatus`：连载状态；
- `observedAt`：数据提供商实际采集数据的时间。

小说信息分为：

- 基础数据；
- 热数据；
- 协议状态。

基础数据包括：

- 首发平台；
- 首发平台书籍ID；
- 书名；
- 作者名；
- 作者在首发平台的ID；
- 阅读链接列表；
- 封面链接；
- 简介；
- 男频、女频或其他频道。

热数据包括：

- 字数；
- 最新章节；
- 来源平台显示的最近更新时间；
- 连载状态；
- 数据采集时间。

协议状态包括：

- `Active`：正常状态；
- `SuspectedSpam`：疑似刷量；
- `Frozen`：刷量确认或被冻结；
- `Duplicate`：重复条目；
- `Invalid`：无效条目；
- `Merged`：已经合并到其他小说；
- `SourceRemoved`：来源平台已经删除或下架；
- 其他协议治理状态。

数据提供商可以提交来源平台展示的基础数据和热数据，但不能自行改变刷量确认、冻结和处罚等协议治理状态。

**小说内部标识**

本协议内部使用递增数字作为小说唯一标识：

`BookID = 1, 2, 3, 4, ...`

BookID由小说登记AO Process统一分配。

Process状态示例：

```
    NextBookID = 1
```

```
    Books = {
        [1] = {
            firstPublishPlatformID = "qidian",
            firstPublishPlatformBookID = "1031940621",
            title = "小说名称",
            authorName = "作者名称",
            authorID = "123456",
            status = "Active"
        }
    }
```

新小说通过基本检查后：

1. 读取当前`NextBookID`；
2. 将该数字分配给小说；
3. 保存小说信息；
4. 建立平台书籍索引；
5. 将`NextBookID`加一。

BookID一旦分配：

- 永久不变；
- 不重复使用；
- 小说被冻结、标记无效或来源平台删除后也不回收；
- 小说被合并后，旧BookID继续保留并指向主BookID；
- 错误入库的BookID也不能重新分配给其他小说。

递增BookID只表示入库顺序，不代表小说质量、热度或排名。

**首发平台和阅读链接**

不单独记录首发链接，只记录首发平台。

首发平台使用：

`firstPublishPlatformID`

小说在首发平台的书籍ID使用：

`firstPublishPlatformBookID`

所有已知阅读页面统一记录在`readURLs`中。

一本小说可以有多个阅读链接：

```
readURLs = [
        {
            platformID = "qidian",
            platformBookID = "1031940621",
            url = "https://...",
            type = "official",
            status = "active"
        },
        {
            platformID = "qqread",
            platformBookID = "456789",
            url = "https://...",
            type = "official",
            status = "active"
        }
    ]
```


每个阅读链接可以记录：

- `platformID`：阅读页面所属平台；
- `platformBookID`：小说在该平台的书籍ID；
- `url`：阅读链接；
- `type`：链接类型，例如官方、作者授权或其他允许类型；
- `status`：链接状态，例如有效、失效或已下架。

首发平台对应的阅读链接也必须包含在`readURLs`中。

如果首发平台的阅读页面失效：

- `firstPublishPlatformID`保持不变；
- 对应阅读链接标记为失效或已下架；
- 其他阅读链接仍可继续使用；
- 不改变小说的BookID。

同一本小说增加其他阅读来源时，只增加`readURLs`记录，不创建新的BookID。

阅读链接不物理删除。

Process应保留：

- 链接首次加入时间；
- 提交该链接的数据提供商；
- 链接失效时间；
- 失效原因；
- 链接恢复时间；
- 对应AO消息ID。

**平台书籍索引**

虽然协议内部使用递增BookID，但仍然需要保存小说在各个平台的书籍ID。

Process为每个阅读平台建立索引：

```
PlatformBookIndex["qidian:1031940621"] = 125681
    PlatformBookIndex["qqread:456789"] = 125681
```


索引键为：

`PlatformBookKey = platformID + ":" + platformBookID`

如果新增小说提交的任一平台书籍ID已经绑定某个BookID，Process不能直接创建新的BookID。

应返回重复提示，由数据提供商判断：

- 是否为同一本小说的新阅读来源；
- 是否为重复条目；
- 是否确实属于不同小说；
- 是否需要进行条目合并或纠错。

AO Process只能自动发现平台书籍ID完全相同的直接重复。

以下复杂情况由数据提供商按照公开规则判断：

- 同一本小说更换了书籍ID；
- 同一本小说更换了链接；
- 同一本书被拆分为多个页面；
- 同一本小说存在新旧版本页面；
- 同一本小说被不同账号重复发布；
- 不同平台转载或联合发布同一本小说。

**入库条件**

最少达到`20,000字`的小说才能正式入库。

对于爬虫支持的平台，由当日数据提供商按照公开规则判断：

- 小说是否达到两万字；
- 小说是否已经入库；
- 是否属于重复条目；
- 是否属于项目支持的小说类型；
- 是否应当创建新小说；
- 是否只是为已有小说增加阅读链接；
- 是否需要修改已有小说信息。

数据提供商不应提交：

- 不足两万字的小说；
- 已经入库的小说；
- 明显重复的小说；
- 不属于小说的页面；
- 无法确定来源的虚假条目；
- 已经被来源平台删除且从未满足入库条件的小说。

数据提供商对入库资格和重复条目作出判断后，将需要入库的小说批量发送给AO Process。

不为每本小说单独进行入库投票或挑战期。

AO Process只进行确定性的基本检查：

- 提交者是否为当日有效数据提供商；
- 首发平台是否为空；
- 首发平台书籍ID是否为空；
- `readURLs`是否为空；
- `readURLs`中是否包含首发平台记录；
- 首发平台记录中的平台书籍ID是否与`firstPublishPlatformBookID`一致；
- 书名是否为空；
- 作者信息是否符合格式；
- 提交字数是否达到两万字；
- 各字段格式是否正确；
- 平台书籍ID是否已经绑定其他BookID；
- 批次是否已经处理；
- 消息是否超过大小或计算量限制。

Process只能检查数据提供商提交的`wordCount`是否达到两万字，不能自行访问来源平台验证实际字数。

数据提供商错误入库不足两万字、重复或虚假的小说，可以通过后续数据提供商纠错或用户质押纠错处理。

小说入库后，即使来源平台后来显示的字数低于两万字，也不自动删除或回收BookID。

可能原因包括：

- 作者删除章节；
- 平台改变字数统计方式；
- 来源平台数据异常；
- 首次入库数据错误。

应根据原因更新实际信息，并决定是否保持正常状态或标记为`Invalid`。

**新小说批量入库**

数据提供商将本次发现的所有新增小说组成批次，发送给小说登记AO Process。

消息示例：

```
{
      "Action": "Add-Books-Batch",
      "JobID": "qidian-2026-06-28",
      "ObservedAt": 1782626400,
      "CrawlerCommit": "git-commit",
      "BatchID": "qidian-2026-06-28-001",
      "Part": 1,
      "TotalParts": 3,
      "Books": [
        {
          "firstPublishPlatformID": "qidian",
          "firstPublishPlatformBookID": "1031940621",
          "title": "小说名称",
          "authorName": "作者名称",
          "authorID": "123456",
          "readURLs": [
            {
              "platformID": "qidian",
              "platformBookID": "1031940621",
              "url": "https://...",
              "type": "official",
              "status": "active"
            },
            {
              "platformID": "qqread",
              "platformBookID": "456789",
              "url": "https://...",
              "type": "official",
              "status": "active"
            }
          ],
          "coverURL": "https://...",
          "description": "小说简介",
          "channel": "male",
          "wordCount": 24563,
          "latestChapter": "第十二章",
          "sourceUpdatedAt": 1782610000,
          "completionStatus": "Serializing"
        }
      ]
    }
```


Process逐条处理批次中的小说。

每条记录可以返回以下结果：

- `Created`：成功入库并分配BookID；
- `AlreadyExists`：某个平台书籍ID已经存在；
- `InvalidFormat`：字段格式错误；
- `BelowMinimumWordCount`：提交字数不足两万字；
- `MissingRequiredField`：缺少必填字段；
- `UnauthorizedProvider`：发送者不是有效数据提供商；
- `Rejected`：其他基本检查未通过。

Process返回批次处理结果：

```
{
      "Action": "Add-Books-Batch-Result",
      "BatchID": "qidian-2026-06-28-001",
      "Created": [
        {
          "firstPublishPlatformID": "qidian",
          "firstPublishPlatformBookID": "1031940621",
          "bookID": 125681
        }
      ],
      "AlreadyExists": [],
      "Rejected": []
    }
```


**批次拆分**

数据提供商不应将无限数量的记录放进同一条AO消息。

协议应限制：

- 每个批次的最大记录数量；
- 每条消息的最大体积；
- 单条简介最大长度；
- 单个字段最大长度；
- 单条消息允许消耗的最大计算量。

超过限制时拆分为多个批次部分：

- `BatchID`：同一批任务的统一标识；
- `Part`：当前部分序号；
- `TotalParts`：总部分数量。

例如：

```
BatchID = "qidian-2026-06-28-new-books"
    Part = 2
    TotalParts = 5
```


每个部分都是独立AO消息。

不要求所有部分同时成功。

Process应分别返回每个部分中成功和失败的记录，数据提供商只需重新提交失败部分。

**批次幂等性**

每个批次必须具有唯一`BatchID`。

Process维护：

```
ProcessedBatches[ProviderAddress][BatchID] = MessageID
```


同一个数据提供商重复提交相同BatchID时，Process不得重复执行。

这样可以避免：

- 网络重试造成重复入库；
- 重复分配BookID；
- 重复更新热数据；
- 同一任务重复领取奖励。

如果需要修正失败批次，应使用新的BatchID，并记录：

- 原BatchID；
- 修正原因；
- 被修正的记录。

**热数据增量更新**

数据提供商只提交发生变化的热数据，不每日重复提交全部小说的热数据。

需要提交的变化包括：

- 字数变化；
- 最新章节变化；
- 来源平台显示的最近更新时间变化；
- 连载状态变化；
- 来源平台删除或下架小说；
- 对以前错误热数据的纠正。

热数据没有变化的小说不放入`Updates`数组。

热数据更新消息示例：

```
{
      "Action": "Update-Hot-Data-Batch",
      "JobID": "all-platforms-2026-06-28",
      "ObservedAt": 1782626400,
      "CrawlerCommit": "git-commit",
      "BatchID": "hot-data-2026-06-28-001",
      "Part": 1,
      "TotalParts": 100,
      "ScannedBooks": 2000000,
      "ChangedBooks": 18532,
      "Updates": [
        {
          "bookID": 125681,
          "wordCount": 356892,
          "latestChapter": "第一百六十二章",
          "sourceUpdatedAt": 1782620000,
          "completionStatus": "Serializing"
        }
      ]
    }
```


每条增量记录只需要包含：

- `bookID`；
- 实际发生变化的字段。

例如只有字数和最新章节变化时：

```
{
      "bookID": 125681,
      "wordCount": 356892,
      "latestChapter": "第一百六十二章"
    }
```


消息级别的`ObservedAt`表示该批数据的实际采集时间。

如果不同记录采集时间差异较大，也可以在单条记录中单独提供`observedAt`。

Process读取小说当前状态：

- 将消息中出现的字段更新为新值；
- 未出现在消息中的字段保持不变。

Process进行以下基本检查：

- BookID是否存在；
- 数据提供商是否有权更新对应平台；
- 字段格式是否正确；
- 字数是否为非负整数；
- 时间格式是否正确；
- 连载状态是否属于允许值；
- 同一批次是否包含重复BookID；
- 批次是否已经处理；
- 数据采集时间是否符合任务周期。

Process应记录每次变化对应的：

- 变化前数值；
- 变化后数值；
- 数据提供商；
- 数据批次；
- 数据采集时间；
- AO消息ID。

**空增量批次**

数据提供商每日必须实际扫描任务范围内的小说。

如果当日没有任何热数据变化，也应提交空增量批次：

```
{
      "Action": "Update-Hot-Data-Batch",
      "JobID": "platform-2026-06-28",
      "ObservedAt": 1782626400,
      "CrawlerCommit": "git-commit",
      "BatchID": "hot-data-2026-06-28-empty",
      "Part": 1,
      "TotalParts": 1,
      "ScannedBooks": 18326,
      "ChangedBooks": 0,
      "Updates": []
    }
```


空增量批次表示数据提供商声称：

- 当日已经执行扫描任务；
- 成功扫描了规定范围；
- 没有发现需要提交的变化。

空增量批次不能证明扫描结果必然正确，仍然可以被后续数据提供商或用户抽查挑战。

以下情况不能提交正常的空增量批次：

- 爬虫请求失败；
- 来源平台返回验证码；
- 页面解析失败；
- 代理IP不可用；
- 本地数据库不可用；
- 只扫描了部分小说；
- 大量页面返回异常内容。

这些情况应提交故障报告。

**增量更新基准**

数据提供商本地可以保存上一轮扫描结果，用于提高比较效率。

但最终应以AO Process当前接受的小说状态作为增量更新基准。

推荐流程：

`读取AO Process当前状态或上一确认批次 → 扫描来源平台 → 比较变化 → 生成增量批次 → 提交AO Process`

不能只与数据提供商自己的本地数据库比较。

否则本地数据库发生错误、回滚或数据丢失时，可能：

- 遗漏需要更新的字段；
- 重复提交旧数据；
- 将旧数据覆盖新数据；
- 生成错误增量。

**基础数据增量修改**

书名、作者名、简介、封面、频道和阅读链接等基础信息发生变化时，也采用增量修改。

消息示例：

```
{
      "Action": "Update-Book-Info-Batch",
      "JobID": "qidian-2026-06-28",
      "ObservedAt": 1782626400,
      "CrawlerCommit": "git-commit",
      "BatchID": "book-info-2026-06-28-001",
      "Updates": [
        {
          "bookID": 125681,
          "title": "修改后的小说名称",
          "coverURL": "https://..."
        }
      ]
    }
```


只有发生变化的字段需要提交。

未出现在消息中的字段保持不变。

**阅读链接更新**

增加阅读链接时：

```
{
      "Action": "Update-Book-Info-Batch",
      "BatchID": "book-info-2026-06-28-001",
      "Updates": [
        {
          "bookID": 125681,
          "addReadURLs": [
            {
              "platformID": "qqread",
              "platformBookID": "456789",
              "url": "https://...",
              "type": "official"
            }
          ]
        }
      ]
    }
```


将链接标记为失效时：

```
{
      "bookID": 125681,
      "disableReadURLs": [
        {
          "platformID": "qqread",
          "platformBookID": "456789",
          "url": "https://..."
        }
      ]
    }
```


恢复链接时：

```
{
      "bookID": 125681,
      "restoreReadURLs": [
        {
          "platformID": "qqread",
          "platformBookID": "456789",
          "url": "https://..."
        }
      ]
    }
```


Process增加阅读链接前应检查：

- URL格式是否合法；
- 同一链接是否已经存在；
- `platformID + platformBookID`是否已经绑定其他BookID；
- 数据提供商是否具有对应平台的更新权限。

阅读链接失效时，不从历史数据中删除，只修改当前状态。

**AO消息和Arweave存储**

新增小说、基础数据修改和热数据增量都直接批量发送给AO Process。

AO消息本身会存储到Arweave，因此不需要再将相同数据单独上传一遍Arweave。

不再要求数据提供商：

- 先上传独立Arweave数据文件；
- 再把文件交易ID发送给AO Process；
- 再保存一份相同的每日完整快照。

完整数据必须直接包含在AO消息中。

不能只向Process提交临时服务器链接，否则永久保留的只是链接，不是链接指向的数据。

AO Process当前状态用于快速查询：

- 当前基础信息；
- 当前字数；
- 当前最新章节；
- 当前连载状态；
- 当前阅读链接；
- 当前协议状态。

历史AO消息用于审计：

- 哪个数据提供商提交了数据；
- 数据在什么时间提交；
- 哪些字段发生变化；
- Process是否接受；
- 数据之后是否被纠正；
- 某个BookID是何时分配的。

**连载状态**

数据提供商可以提交来源平台能够直接观察到的状态，例如：

- `Serializing`：连载中；
- `Completed`：已完结；
- `Paused`：来源平台明确标记暂停；
- `Removed`：来源页面已经删除或下架；
- `Unknown`：暂时无法确定。

“已太监”通常不是来源平台提供的客观状态，不能仅因为长期未更新就由数据提供商直接写入。

长期没有更新的小说可以根据`sourceUpdatedAt`在前端显示：

`长期未更新`

只有满足以下条件之一时，才能将小说标记为已太监：

- 来源平台明确标记；
- 作者明确声明停止创作；
- 用户提交可验证证据并通过相应流程；
- 协议治理按照明确标准确认。

小说第一次变为`Completed`时，Process记录：

- 完结状态；
- 完结时字数；
- 最终章节；
- 来源平台显示的更新时间；
- 数据提供商采集时间；
- 对应AO消息ID。

完结后的小说不需要每日重复提交热数据。

只有发生以下情况时再次提交：

- 发布番外；
- 重新连载；
- 字数或章节变化；
- 来源信息修改；
- 来源页面删除；
- 数据纠错。

**重复小说处理**

数据提供商在入库前负责按照规则排除重复小说。

如果入库后发现两个BookID实际属于同一本小说，不物理删除任何BookID，而是执行合并：

```
Books[DuplicateBookID].status = "Merged"
    Books[DuplicateBookID].mergedInto = CanonicalBookID
```


合并后：

- 主BookID继续正常使用；
- 重复BookID停止接受普通操作；
- 查询重复BookID时跳转到主BookID；
- 历史数据继续保留；
- 重复BookID永不重新分配；
- 已有标签、评论、评分和助资按照专门的合并规则处理。

数据提供商可以主动提交重复条目合并请求。

用户也可以通过质押体系指出数据提供商遗漏的重复条目。

**数据纠错**

数据提供商发现自己提交错误数据后，可以主动提交纠错批次。

纠错消息应包括：

- BookID；
- 发生错误的字段；
- 正确字段；
- 原BatchID；
- 纠错原因；
- 新的数据采集时间。

主动纠正轻微且非故意的错误，可以只记录错误次数，不一定罚没全部保证金。

以下情况可以处罚：

- 错误已经影响用户操作；
- 数据提供商发现错误后长期不纠正；
- 大量记录出现相同错误；
- 数据提供商隐瞒已知爬虫故障；
- 提交明显未经扫描的虚假批次；
- 伪造来源数据；
- 伪造数据采集时间。

历史错误消息不会从Arweave删除。

Process只更新当前有效状态，并记录当前状态由哪次纠错消息产生。

**爬虫支持的平台**

起点、番茄等能够通过项目统一爬虫稳定获取数据的平台，由数据提供商维护。

数据提供商每日负责：

- 扫描任务范围内的来源平台；
- 判断新增小说是否达到两万字；
- 排除已经存在或明显重复的小说；
- 批量提交新增小说；
- 提交发生变化的基础数据；
- 提交发生变化的热数据；
- 没有变化时提交空增量批次；
- 报告来源平台或爬虫故障；
- 使用项目允许的爬虫版本；
- 接受后续纠错和抽查。

项目应公开维护统一的爬虫代码。

统一维护爬虫的原因包括：

- 降低新数据提供商参与成本；
- 避免每个数据提供商重复开发同类爬虫；
- 方便更换数据提供商；
- 统一字段和数据格式；
- 方便复现和检查爬取结果；
- 降低单个数据提供商形成技术垄断的风险。

数据提供商可能需要自行购买：

- 服务器；
- 代理IP；
- 其他运行资源。

这些成本应包含在任务报价中，或者按照任务提案明确区分为成本报销。

**不支持爬虫的平台**

暂未被统一爬虫支持的平台，通过质押体系允许任何人登记和修改小说基础信息。

用户提交：

`新增小说请求 + 小说信息 + 质押保证金`

请求经过挑战期后，由小说登记AO Process：

- 进行基本检查；
- 分配递增BookID；
- 保存小说信息；
- 退还提交者保证金。

手动入库不获得额外登记奖励。

提交者通常是为了继续对小说进行：

- 评分；
- 评论；
- 打标签；
- 加入书单；
- 提供助资；
- 参与其他平台功能。

如果为手动入库提供直接奖励，容易导致用户：

- 批量登记无人使用的小说；
- 登记不足两万字的小说；
- 登记重复小说；
- 伪造不存在的小说；
- 将一本小说拆分为多个条目。

**小众平台热数据**

暂未被项目爬虫支持的小众平台，不要求用户持续手动更新热数据。

这些小说可以保留：

- 最后一次字数；
- 最后一次最新章节；
- 最后一次来源平台更新时间；
- 最后一次验证时间。

前端应明确显示：

- 当前没有持续数据提供商；
- 热数据可能已经过期；
- 最后一次有效更新时间。

没有数据提供商时，不自动更新或推测：

- 字数；
- 最新章节；
- 最近更新时间；
- 连载状态。

小众平台使用量达到一定规模后，可以通过治理：

- 开发该平台的统一爬虫；
- 将该平台加入数据提供商任务；
- 为持续更新提供预算。

**数据提供商竞标**

所有满足资格要求的人都可以质押参与数据提供商竞标。

项目早期可以每日选择一个数据提供商，负责：

- 入库所有爬虫支持平台的新增小说；
- 更新所有爬虫支持平台小说的基础数据变化；
- 更新所有爬虫支持平台小说的热数据变化。

随着数据规模增加，可以按照以下方式拆分任务：

- 每个平台一个任务；
- 多个小平台组成一个任务；
- 大型平台拆分为多个任务；
- 按照小说ID范围拆分任务。

竞标者提交：

- 要求的任务报酬；
- 数据提供商保证金；
- 可以执行的平台范围；
- 使用的钱包地址；
- 承诺使用的爬虫版本。

符合资格和最低保证金要求的竞标者中，原则上由要求报酬最低者中标。

为避免竞标者复制其他人的报价后只降低极小金额，可以采用密封竞标：

`提交报价哈希 → 报价截止 → 公开报价 → 确定数据提供商`

不能只比较报价，还应要求：

- 最低保证金；
- 爬虫运行测试；
- 历史任务成功率；
- 历史错误率；
- 当前是否存在未处理违规。

**数据提供商保证金**

数据提供商需要锁定保证金。

可以采用：

$ProviderBond = \max(MinProviderBond,\ ProviderReward \times BondMultiplier)$

其中：

- `ProviderBond`：数据提供商保证金；
- `MinProviderBond`：最低保证金；
- `ProviderReward`：本次数据任务报酬；
- `BondMultiplier`：保证金倍率。

保证金用于约束：

- 未按时提交数据；
- 提交空批次冒充完成扫描；
- 大量遗漏新增小说；
- 批量提交错误热数据；
- 提交重复或虚假小说；
- 使用未声明的爬虫版本；
- 隐瞒爬虫故障；
- 拒绝纠正已知错误。

保证金不能在任务提交后立即全部解锁，应保留纠错期，使后续用户或下一任数据提供商可以发现历史错误。

**数据提供商故障**

数据提供商未在规定时间提交任务时：

1. 进入短暂宽限期；
2. 宽限期结束后开放其他合格数据提供商接管；
3. 接管者完成任务后获得当日任务报酬；
4. 原数据提供商不获得当日奖励；
5. 根据未完成原因决定是否罚没保证金。

如果来源平台本身发生故障，数据提供商应提交故障报告，而不是提交虚假的正常数据。

故障报告应包括：

- 受影响平台；
- 故障开始时间；
- 受影响范围；
- 错误类型；
- 可验证证据；
- 已经尝试的解决方式；
- 是否需要修改爬虫。

来源平台真实故障得到确认后，不将无法获取数据直接视为数据提供商作弊。

但以下行为仍应处罚：

- 没有实际运行爬虫却声称任务完成；
- 明知数据为空仍提交正常状态；
- 隐瞒大规模字段解析错误；
- 使用旧数据伪装成当日数据；
- 伪造数据采集时间；
- 只扫描部分小说却声称已经完整扫描。

**爬虫Bug和修复**

应鼓励数据提供商和其他开发者公开提交：

- 爬虫Bug；
- 页面结构变化；
- 字段解析错误；
- 小说遗漏问题；
- 性能问题；
- 修复代码；
- 测试用例。

重要爬虫Bug的报告和修复奖励，应高于数据提供商正常运行数日获得的劳动报酬。

否则数据提供商可能更有动力只修复自己的本地版本，而不向项目公开修复。

关键Bug修复奖励可以满足：

$CriticalFixReward \ge \max(VerifiedLaborReward,\ 3 \times AverageDailyProviderLaborReward)$

其中：

- `CriticalFixReward`：关键Bug修复奖励；
- `VerifiedLaborReward`：经确认的有效修复劳动报酬；
- `AverageDailyProviderLaborReward`：该数据任务近期平均每日劳动报酬。

普通、小范围或重复Bug不适用三日最低奖励，应根据实际影响和劳动量计算。

数据提供商提交批次时必须声明：

- 爬虫版本；
- Git Commit；
- 数据格式版本；
- 必要的配置版本。

如果使用本地修改后的爬虫：

- 必须公开补丁；
- 必须说明修改原因；
- 必须提交测试结果；
- 不能长期使用未公开的私有版本执行正式任务。

使用未声明的私有版本，可以作为任务不合格或处罚理由。

**数据提供商奖励和声誉**

数据提供商通过竞标获得每日任务报酬。

任务报价应尽量区分：

- 劳动报酬；
- 服务器费用；
- 代理IP费用；
- 其他实际成本。

只有劳动报酬部分生成所属DAO声誉。

例如：

- 劳动报酬：`120代币`；
- 服务器和代理费用：`80代币`；
- 其他成本：`20代币`；
- 总付款：`220代币`。

数据提供商获得：

- `220代币`总付款；
- `120点`小说登记DAO或所属内容DAO声誉。

成本报销部分不生成声誉。

数据任务只有满足以下条件后才支付：

- 在期限内完成；
- AO Process成功接受批次；
- 完成规定的扫描范围；
- 没有提交虚假完成声明；
- 没有成立的数据造假挑战；
- 完成规定的纠错期或挑战期。

**最终流程**

爬虫支持平台的新小说入库流程：

`数据提供商运行统一爬虫 → 判断小说达到2万字且不是重复条目 → 批量发送给AO Process → Process基本检查 → 分配递增BookID → 保存小说信息`

基础数据更新流程：

`数据提供商扫描来源平台 → 发现书名、简介、封面或阅读链接变化 → 生成增量修改 → 批量发送给AO Process → Process更新当前状态`

热数据更新流程：

`数据提供商扫描全部任务范围 → 与AO Process当前状态比较 → 只生成发生变化的记录 → 批量发送增量 → Process更新当前热数据 → AO消息永久保留变化历史`

没有热数据变化时：

`完成扫描 → 提交空增量批次 → Process记录当日任务声明`

不支持爬虫的平台：

`用户质押提交小说 → 挑战期 → Process基本检查 → 分配BookID → 保存小说信息 → 退还保证金`

错误数据处理：

`数据提供商主动纠错，或用户质押指出错误 → 验证证据 → 更新Process当前状态 → 保留历史消息 → 根据责任处理保证金`

重复小说处理：

`发现重复条目 → 提交合并请求 → 确认主BookID → 重复BookID标记为Merged → 保留历史并跳转到主BookID`

基本原则：

- 数据提供商负责按照公开规则判断哪些小说满足入库条件；
- 数据提供商负责在入库前排除重复条目；
- AO Process只负责基本检查、递增分配BookID和维护当前状态；
- 所有阅读页面统一存入`readURLs`；
- 只记录首发平台，不单独记录首发链接；
- 数据提供商只提交热数据增量；
- 没有变化时仍需提交空增量批次；
- 新增小说和数据更新直接批量发送给AO Process；
- AO消息保存完整数据和历史变化，不重复上传相同数据到Arweave；
- Process当前状态用于快速查询，历史AO消息用于审计和重建。

## 小说状态

每本小说有以下状态：

- Active：正常状态，可以提交普通标签，可以领取标签奖励。
- SuspectedSpam：疑似刷量状态，有人发起刷量指控，暂停新增标签奖励，等待裁决。
- ConfirmedSpam / Frozen：刷量确认或冻结状态，禁止领取普通标签奖励，普通标签提交被禁用或仅允许无奖励纠错。
- Restored：复审成功后恢复正常状态。

状态流转：

- 新登记小说默认为 Active。
- 有人高质押发起刷量指控后，进入 SuspectedSpam。
- 刷量指控成立后，进入 ConfirmedSpam / Frozen。
- 刷量指控失败后，恢复 Active。
- Frozen 状态下可以通过高质押复审恢复为 Restored / Active。

只有 `Active` 或 `Restored` 状态能进行普通评论、评分、打标签等操作。

普通评论不需要质押，也不视为评论者对小说正常性的经济担保。用户第一次针对该小说提交需要质押或领取协议奖励的操作时，才默认担保该小说不是刷量小说。若该小说之后被确认刷量，只能按照相应操作的规则处理其仍处于锁定状态的保证金、未释放奖励或明确承担正常性风险的质押，不得处罚仅发表普通评论的用户。

小说刷量确认后：

- 进入 Frozen；
- 禁止其他质押复审外的操作；
- 不进入公共覆盖池；
- 不进入推荐、排行、补贴队列；
- 已有标签保留但标记为冻结或无奖励；
- 未释放标签奖励被追回；
- 第一个质押操作者承担部分正常性风险；
- 该小说相关质押金、未释放奖励按规则分配。

已有标签处理：

- 已确认准确标签：可以保留，但不再产生奖励；
- 未确认标签：冻结；
- 明显配合刷量的标签：删除或标为无效。

## 助资池

每本小说可以拥有独立助资池：

```

小说助资池
    ├─ 标签奖励额度
    ├─ 评分奖励额度
    ├─ 阅读任务额度
    └─ 其他用途额度
```

- 用户可以向一本小说提供助资。
- 小说助资者可以指定助资用途。
- 只有明确允许用于标签奖励的小说助资，才能参与标签奖励计算。
- 小说助资池应分别记录不同用途的余额，避免标签奖励消耗原本用于评分、阅读任务或预测市场的资金。

小说助资者可以选择：

- 全部用于标签奖励；
- 按一定比例用于标签奖励；
- 完全不用于标签奖励；
- 交由小说助资池的治理规则决定用途；
- 作为不限用途助资，由协议按照预设比例分配。

未明确允许用于标签奖励的助资，不计入标签奖励公式。

如果助资者选择不限用途，可以由协议预设比例或相关治理提案，将其中一部分划入小说的标签助资余额。

# 刷量指控

刷量指控适用场景：

- 小说明显为刷奖励批量创建；
- 小说内容或元数据明显伪造；
- 同一本书被拆成多个条目；
- 重复登记；
- 作者名、平台来源、简介等关键字段明显造假；
- 该小说与批量垃圾条目存在明显模式关联。

刷量指控流程：

- 举报者质押 SpamFlagBond。
- 小说进入 SuspectedSpam。
- 暂停对该小说的其他操作。
- 进入挑战期和裁决期。
- 指控成立，小说进入 Frozen。
- 指控失败，小说恢复 Active，举报者质押被罚没。

SpamFlagBond 建议：
SpamFlagBond = max(基础刷量指控质押, 该小说过去 30 天标签奖励总额 × 2)
如果小说还没有任何标签或奖励记录，则使用基础刷量指控质押。

未打标签小说的刷量清理奖励
为了避免用户浪费时间阅读刷量小说，允许用户在小说没有任何普通标签时发起刷量指控
如果指控成功，举报者获得小额清理奖励。

清理奖励可以来自两个来源：

- 如果该小说有助资，则优先从助资中支付；
- 如果该小说没有助资，则只从协议 CleanupPool 支付极小额奖励。

未打标签阶段举报成功奖励只给极小额清理奖励，例如 1-3 TOKEN，且严格受每日预算限制。

## 协议小额清理奖励

采用协议提供小额清理奖励，设置封顶。
规则：

- 全平台每日 CleanupPool 封顶；
- 单作者每日可触发清理奖励次数封顶；
- 单来源平台每日可触发清理奖励次数封顶；
- 单本小说只能领取一次清理奖励；
  示例：
- 全平台每日清理预算：100 TOKEN；
- 单次未打标签垃圾小说举报奖励：1-3 TOKEN；
- 单作者每日最多触发 2 次；
- 单平台每日最多触发 20 次。

清理奖励只是“清洁补贴”，不能成为主要收益来源。

单本基础奖励：

- 同一作者前几本给小额奖励；
- 同一作者触发过多后，单本奖励递减或停止。

示例：

- 同一作者第 1 本确认刷量：3 TOKEN；
- 第 2 本：2 TOKEN；
- 第 3 本：1 TOKEN；
- 后续不再给单本协议奖励。

## 作者级风险继承

作者风险不能因为一本书刷量就直接永久定罪。
应采用分级风险继承。
作者状态：

- Normal：正常作者。
- Risk-1：已有 1 本小说确认刷量。
- Risk-2：多本小说确认刷量或刷量比例较高。
- SpamAuthor：系统性刷量作者。

规则：

- 1 本确认刷量：

  - 作者进入 Risk-1；
  - 该作者其他书进入风险提示状态；
  - 暂停或降低公共标签奖励；
  - 不直接标为刷量确认。
- 30 天内 ≥ 3 本确认刷量，或确认刷量比例 ≥ 30%：

  - 作者进入 Risk-2；
  - 该作者其他未验证小说默认无奖励；
  - 普通标签 TagBond 提高。
- ≥ 5 本确认刷量，或确认刷量比例 ≥ 70%：

  - 作者进入 SpamAuthor；
  - 该作者所有未验证小说默认 Frozen；
  - 新登记必须走高质押申诉或复审流程。

## 平台级风险继承

如果某小说来源平台大量小说被确认刷量，则平台进入风险状态。

平台风险指标：
PlatformRisk = ConfirmedSpamBooks / AuditedBooks

注意：

- 分母应使用已审计、已挑战、已进入奖励流程的小说数；
- 不直接使用平台总书数；
- 必须设置最低样本数门槛。

建议规则：

- 样本数 ≥ 30 且刷量比例 ≥ 30%：

  - 平台进入 Watchlist；
  - 新书降低标签奖励优先级；
  - TagBond 提高。
- 样本数 ≥ 50 且刷量比例 ≥ 50%：

  - 平台进入 Quarantined；
- 样本数 ≥ 100 且刷量比例 ≥ 70%：

  - 平台进入 Blocked；
  - 暂停登记该平台新书；
  - 已登记数据保留；
  - 允许治理或高质押复审恢复。

平台级处理不应是“一次发现大量刷量就永久拉黑”，而应是分级隔离和可复审。

## 作者级 / 平台级模式奖励

为了避免用户逐本举报垃圾小说薅协议奖励，可以设置“模式发现奖励”。

清理奖励分为：

- 单本基础清理奖励；
- 作者级模式奖励；
- 平台级模式奖励。

作者级模式奖励：

- 当同一 author_key 下 ≥ 3 本小说被确认刷量；
- 给最早提交作者级模式证据的人一次性奖励；
- 建议 10-30 TOKEN；
- 只发一次。

平台级模式奖励：

- 当同一 platform_key 下确认刷量比例超过阈值，且样本数达标；
- 给最早提交平台级模式证据的人一次性奖励；
- 建议 50-200 TOKEN；
- 需要更高质押和更长挑战期；
- 只发一次。

## 指控质押分级

影响范围越大的指控，需要越高质押。

## 刷量指控失败后的处理

如果刷量指控失败：

- 小说恢复 Active；
- 举报者 SpamFlagBond 被罚没；
- 一部分补偿受影响的登记者和标签者；
- 一部分进入 AuditPool；
- 暂停期间被影响的标签奖励可以恢复；
- 恶意反复举报者可降低权限或提高后续质押要求。

## 复审机制

为了避免误判，Frozen 小说可以复审。

复审规则：

- 复审需要高质押；
- 复审期可以提交新证据；
- 复审成功后恢复 Active / Restored；
- 复审失败，复审质押被罚没；
- 同一本小说不能无限频繁复审，需设置冷却期。

# 标签

为什么要给小说打标签

- 筛选书籍。一本书值不值得看，无非就是优缺点的权衡，优点（爽点）是不是足够吸引你读下去，缺点（毒点）是不是足够劝退你，两者一对比，心里就有数了。但各小说平台的标签不太准确，且没有全面的毒点标签。
- 悬赏定制符合指定标签组合的书籍

所以标签功能的目标是：

- 鼓励用户给每本正常小说打尽可能全面正确的标签，但鼓励优先打感兴趣的用户更多的标签
- 同时要避免用户在各个小说平台创建大量没人看的小说，然后在本平台打标签刷奖励

打标签方式：

- 用户质押代币打标签获得奖励
- 只能对2万字数以上的小说打标签

注意：

- 垃圾小说可以存在，但给它打标签没有正收益。

## 标签类别

标签的定义应尽量清晰，不能有太大争议，否则将浪费大量时间在裁决一本小说符不符合某个标签上。

标签种类与标签名称不固定，缺少的标签种类与标签名称用户可以质押代币添加，具体奖励与惩罚规则你完善下

增加标签名称时要标明此标签的互斥标签，因为互斥标签不能同时标记到同一本小说。

标签举例：
**爽点**
杀伐果断、题材新颖(反向标签套路陈旧、同义标签反套路)、文笔在线、多女主、布局流、腹黑、逻辑(反向标签前后矛盾)

**毒点**
男同、套路陈旧、文笔差、畏畏缩缩(犹犹豫豫)、送女、失忆、幕后流、群像、领主类主角亲自下场打打杀杀、挂太大、冒险、前后矛盾(AI写作易出现)
舔狗(舔异性，舔名人)

有的标签对一些读者来说是爽点，对另外一些读者来说可能是毒点，本项目不直接对此进行区分，用户筛选书籍时根据自己口味指定标签即可，这里的爽点与毒点仅用于举例。

## 提交标签

用户提交普通标签时，需要质押 TagBond。
TagBond = max(s_min, m × TagReward)
建议参数：

- s_min：最低质押，例如 0.1-20 TOKEN。
- m：质押倍率，例如 2-5。
- TagReward：该标签当前可领取的奖励。

用户提交标签后：

- 标签进入挑战期。
- 挑战期内，其他用户可以质押挑战该标签。
- 如果标签通过，TagBond 解锁，奖励全部释放。
- 如果标签被挑战成功，TagBond 被罚没，标签无效，奖励不发。
- 如果标签所在小说在挑战期内被确认刷量，奖励不发，提交者承担部分罚没。

普通标签提交者默认承担两类风险：

- 标签准确性风险：标签错误会被挑战并罚没。
- 小说正常性风险：小说被确认刷量、伪造、重复、恶意拆分时，相关未释放奖励会被追回，近期参与者可能承担额外罚没。
  “刷量”不是普通内容标签，而是特殊状态标签，会影响整本小说是否还能继续获得标签奖励

## 标签奖励

奖励来源

- 协议通胀
- 该小说的助资池

**整体结构**

- 内容DAO下设一个统一的标签DAO。
- 标签DAO拥有独立资金池、独立声誉、正式成员、专业治理委托和标签争议处理机制。
- 每个具体标签不单独成立DAO。
- 每个规范标签只在标签DAO统一资金池中拥有一个虚拟余额。
- 标签奖励还可以来自对应小说的助资池。
- 所有真实代币统一由标签DAO奖励合约和小说助资合约保管，标签虚拟余额只用于内部记账。

结构示例：

```

内容DAO
    └─ 标签DAO
       ├─ 标签奖励资金池
       │  ├─ 第四天灾：虚拟余额
       │  ├─ 系统流：虚拟余额
       │  ├─ 群像：虚拟余额
       │  └─ 其他规范标签：虚拟余额
       ├─ 标签挑战与清理预算
       └─ 标签系统维护预算
```

基本原则：

`标签DAO是真正的DAO，具体标签只是标签DAO内部的奖励账户。`

`标签余额支持该标签在所有小说上的数据建设，小说助资余额支持该小说的标签数据建设。`

**标签资金分配**

- 持币者可以将内容DAO中的资金分配票继续分配给标签DAO。
- 分配给标签DAO的资金票，可以继续指定具体标签。
- 用户关注多个标签时，其标签资金分配票默认在所关注的标签之间平均分配。
- 用户可以手动设置具体标签的资金分配比例。
- 用户也可以只关注某个标签，但不向其分配资金票。
- 建议在界面中区分：
  - 关注标签：用于订阅、筛选和推荐；
  - 资助标签：用于决定标签奖励资金的流向。
- 用户关注标签时可以默认同时资助，但允许单独关闭资助或修改比例。
- 没有继续分配给具体标签的票，对应资金留在标签DAO本级。

标签DAO本级资金可以用于：

- 错误标签挑战奖励；
- 标签清理；
- 标签合并和废弃；
- 标签定义维护；
- 标签系统开发和维护；
- 跨标签公共任务；
- 其他无法归属于单一标签的支出。

某标签当日获得的新增虚拟余额为：

$$
TagAllocation
=
TagDAORewardBudget
\times
\frac{TagVotes}{TotalTagVotes}

$$

其中：

- `TagDAORewardBudget`：标签DAO当日用于具体标签奖励的新增资金；
- `TagVotes`：该标签获得的有效资金分配票；
- `TotalTagVotes`：所有具体标签获得的有效资金分配票总数；
- `TagAllocation`：该标签当日增加的虚拟余额。

标签虚拟余额变化为：

$$
NewTagRemain
=
OldTagRemain
+
TagAllocation
-
LockedTagDebit
+
ReturnedTagDebit

$$

其中：

- `OldTagRemain`：计算前该标签可用虚拟余额；
- `LockedTagDebit`：新提交标签时从该标签余额锁定的奖励金额；
- `ReturnedTagDebit`：标签挑战成功后退回该标签的金额；
- `NewTagRemain`：计算后的标签可用虚拟余额。
- 标签虚拟余额不是独立国库。
- 标签不能独立转账。
- 标签创建者不能提取标签虚拟余额。
- 标签创建者不会自动获得该标签的治理权。
- DAO国库和其他协议控制地址持有的代币不产生标签资金分配票。

**新标签**

- 新标签不设置启动额度。
- 新标签成为规范标签后，其标签虚拟余额初始值为 `0`。
- 新标签没有获得持币者资金分配票时，仍然可以正常使用。
- 即使具体标签余额为零，如果对应小说存在标签助资余额，添加该标签仍可能获得奖励。
- 如果具体标签余额和小说标签助资余额都为零，添加标签不产生代币奖励。
- 不因为标签较新、较冷门或使用人数较少而提供额外协议补贴。

新标签成立流程可以为：

`提交标签定义 + 质押保证金 → 挑战期 → 无有效挑战 → 成为规范标签`

新标签提案应说明：

- 标签名称；
- 标签定义；
- 适用条件；
- 不适用条件；
- 示例小说；
- 与现有标签的区别；
- 同义词和别名；
- 可能存在的互斥标签；
- 所属标签类别。

如果新标签被认定为已有标签的同义词：

- 不建立独立虚拟余额；
- 新名称作为规范标签的别名；
- 用户对别名的搜索、关注和资助映射到规范标签；
- 所有资金分配票统一计入规范标签；
- 同一本小说不能通过规范标签和别名重复获得奖励。

**标签奖励资金来源**

定义：

- `TagRemain`：该规范标签当前尚未锁定的可用虚拟余额；
- `BookTagRemain`：该小说助资池中当前尚未锁定、且明确可以用于标签奖励的余额；
- `TotalRemain`：本次标签奖励计算可以使用的总余额。

总余额为：

$$
TotalRemain
=
TagRemain
+
BookTagRemain

$$

两种资金来源分别发挥不同作用：

- `TagRemain`支持该标签在所有小说上的数据建设；
- `BookTagRemain`支持该小说的所有有效标签建设；
- 两者共同决定一次“小说—标签”关系的奖励。

因此：

- 某个标签获得较多资金支持时，在不同小说上添加该标签都可能获得较高奖励；
- 某本小说获得较多标签助资时，为该小说添加不同的有效标签都可能获得较高奖励；
- 某个冷门标签即使自身余额较低，如果对应小说获得较多标签助资，仍可以产生合理奖励；
- 某本小说即使没有助资，如果添加的是获得较多标签资金支持的标签，仍可以获得奖励。

**标签奖励参数**

标签奖励使用以下治理参数：

- `RewardDivisor`：奖励除数；
- `R_max`：全局单次标签奖励上限。

初始参数：

`RewardDivisor = 500`

- `RewardDivisor`越小，相同余额下的单次奖励越高，资金消耗越快。
- `RewardDivisor`越大，相同余额下的单次奖励越低，资金能够支持的标注次数越多。
- `RewardDivisor`可以由标签DAO治理修改。
- `R_max`可以由标签DAO治理修改。
- 参数修改属于标签DAO的重要治理事项，应由标签DAO专业治理决定，并保留全员复决窗口。
- 参数修改只影响生效后的新标签提交。
- 已经计算并锁定的标签奖励不因参数修改而重新计算。
- 参数修改应设置公开生效延迟，避免治理结果生效前出现突击提交。

**标签奖励公式**

本次有效添加标签获得的代币奖励数量统一记为：

`TagReward`

标签奖励公式为：

$$
TagReward
=
\min
\left(
R_{\max},
\frac{TagRemain+BookTagRemain}{RewardDivisor}
\right)

$$

使用总余额表示为：

$$
TagReward
=
\min
\left(
R_{\max},
\frac{TotalRemain}{RewardDivisor}
\right)

$$

按照初始参数表示为：

$$
TagReward
=
\min
\left(
R_{\max},
\frac{TagRemain+BookTagRemain}{500}
\right)

$$

其中：

- `TagReward`：本次标签提交通过后获得的代币奖励数量；
- `TagRemain`：该标签当前尚未锁定的可用虚拟余额；
- `BookTagRemain`：该小说当前尚未锁定的标签助资余额；
- `RewardDivisor`：奖励除数，由标签DAO治理修改，初始值为 `500`；
- `R_max`：全局单次标签奖励上限。

不设置全局最低奖励。

因此：

- 标签奖励可以无限接近于零；
- 当两个资金来源都为零时，标签奖励为零；
- 奖励为零时，用户仍可以正常添加标签；
- 协议不形成待支付欠款；
- 不从其他标签的余额借款；
- 不从其他小说的助资池借款；
- 不从标签DAO本级资金自动补足。

当：

$$
TagRemain+BookTagRemain=0

$$

则：

$$
TagReward=0

$$

**全局奖励上限**

- `R_max`用于限制单次标签奖励的最大数量。
- 设置上限可以防止某个长期无人使用的标签或小说助资池积累大量余额后，一次支付过高奖励。
- `R_max`应根据发现、阅读、核验并提交一个有效标签通常需要的合理劳动时间确定。
- 超出 `R_max` 的理论奖励部分不会支付，也不会从任何余额中扣除。

例如：

- `TagRemain = 20,000`
- `BookTagRemain = 30,000`
- `RewardDivisor = 500`
- `R_max = 60`

则：

$$
TagReward
=
\min
\left(
60,
\frac{20000+30000}{500}
\right)

$$

$$
TagReward
=
\min(60,100)
=
60

$$

没有支付的 `40` 不从标签余额或小说助资余额中扣除。

**资金来源的扣款方式**

计算出 `TagReward` 后，按照两个资金来源当前余额的比例承担奖励。

标签余额承担金额为：

$$
TagDebit
=
TagReward
\times
\frac{TagRemain}{TagRemain+BookTagRemain}

$$

小说标签助资余额承担金额为：

$$
BookDebit
=
TagReward
\times
\frac{BookTagRemain}{TagRemain+BookTagRemain}

$$

并满足：

$$
TagDebit+BookDebit=TagReward

$$

其中：

- `TagDebit`：从该标签虚拟余额中锁定的金额；
- `BookDebit`：从该小说标签助资余额中锁定的金额。

例如：

- `TagRemain = 3,000`
- `BookTagRemain = 7,000`
- `TagReward = 10`

则：

$$
TagDebit
=
10
\times
\frac{3000}{10000}
=
3

$$

$$
BookDebit
=
10
\times
\frac{7000}{10000}
=
7

$$

即：

- 标签虚拟余额承担 `3`；
- 小说标签助资余额承担 `7`。

按比例扣款可以避免：

- 固定优先耗尽标签余额；
- 固定优先耗尽小说助资余额；
- 某一个资金来源承担全部奖励，而另一个来源长期不消耗。

如果某一来源余额为零，则全部奖励由另一来源承担。

当：

$$
TagRemain=0

$$

且：

$$
BookTagRemain>0

$$

则：

$$
TagDebit=0

$$

$$
BookDebit=TagReward

$$

当：

$$
BookTagRemain=0

$$

且：

$$
TagRemain>0

$$

则：

$$
BookDebit=0

$$

$$
TagDebit=TagReward

$$

**代币精度和扣款误差**

- `TagReward`、`TagDebit`和`BookDebit`均按照代币最小精度向下取整。
- 不进行四舍五入，避免实际支付超过公式结果。
- 因比例计算产生的最小单位差额，可以由余额较多的一方承担。
- 最终必须满足：

$$
TagDebit+BookDebit=TagReward

$$

- 实际扣款不得超过任一资金来源的当前可用余额。
- 如果计算误差导致扣款超过余额，应重新调整差额。
- 计算结果小于一个代币最小单位时，记为零。
- 奖励为零时，用户仍然可以正常提交标签。

假设代币最小单位为 $10^{-6}$，则奖励向下取整为：

$$
TagReward
=
\frac{
\left\lfloor
TagReward_{\mathrm{raw}}
\times
10^6
\right\rfloor
}{
10^6
}

$$

**奖励计算和锁定**

用户提交新的“小说—标签”关系时：

1. 读取该标签当前可用的 `TagRemain`；
2. 读取该小说当前可用的 `BookTagRemain`；
3. 读取当前生效的 `RewardDivisor`和`R_max`；
4. 根据公式计算 `TagReward`；
5. 根据两个资金来源的余额比例计算 `TagDebit`和`BookDebit`；
6. 按照代币最小精度处理计算结果；
7. 立即从两个资金来源分别锁定对应金额；
8. 标签关系进入挑战期；
9. 挑战期结束后，根据结果支付或退回锁定资金。

流程为：

`提交标签 → 计算TagReward → 计算TagDebit和BookDebit → 分别锁定 → 挑战期 → 确认后支付`

必须在提交时立即计算并锁定奖励，防止多个用户同时按照相同余额重复计算奖励。

锁定后：

$$
NewTagRemain
=
OldTagRemain
-
TagDebit

$$

$$
NewBookTagRemain
=
OldBookTagRemain
-
BookDebit

$$

标签通过挑战期后：

- 锁定的 `TagDebit`和`BookDebit`共同支付给标签提交者；
- 不再重复从余额中扣除；
- 用户获得 `TagReward`枚代币；
- 用户获得与 `TagReward`等额的标签DAO声誉；
- 系统记录用户对该具体标签的历史贡献。

标签挑战成功后：

- 标签关系被删除；
- `TagReward`不支付；
- `TagDebit`退回原标签虚拟余额；
- `BookDebit`退回原小说标签助资余额；
- 不得把退回资金转入另一资金来源；
- 不生成标签DAO声誉；
- 根据规则处理标签提交者的保证金；
- 挑战者可以获得清理奖励。

**标签确认和声誉**

标签添加采用乐观确认：

`用户提交标签 → 进入挑战期 → 无有效挑战 → 自动确认`

普通标签添加不需要标签DAO正式成员逐条审核。

标签确认后：

`1分钟已经确认的标签人类劳动时间 = 获得1点标签DAO声誉`

例如：

- 标签人类劳动时间为 `8.5分钟`
- 用户获得 `8.5枚代币`
- 用户同时获得 `8.5点标签DAO声誉`

声誉数量只根据已经确认的标签人类劳动时间计算，不区分代币奖励来自：

- 标签虚拟余额；
- 小说标签助资余额。

系统还可以分别记录：

- 标签DAO累计声誉；
- 标签DAO活跃声誉；
- 具体规范标签的贡献记录；
- 用户参与过的小说标签建设记录。

具体标签贡献记录可以用于：

- 展示专业经历；
- 判断用户熟悉哪些标签；
- 选择标签争议评审者；
- 识别异常标注行为。

具体标签贡献不形成独立DAO声誉，也不使每个具体标签拥有独立正式成员体系。

**标签有效性规则**

取消按小说已有标签数量降低奖励后，应通过标签有效性规则控制无意义的过度标注。

获得奖励的标签必须：

- 真实符合小说内容；
- 符合规范标签定义；
- 具有独立语义；
- 不是已有标签的同义重复；
- 不是通过细微措辞人为拆分出的重复标签；
- 尚未在该小说上确认；
- 已经通过规定的挑战期；
- 不属于已废弃标签；
- 不属于单纯系统状态或审核状态。

一本小说拥有大量标签本身不构成违规。

只要标签真实、独立并且符合定义，就可以获得公式计算出的奖励。

**重复添加限制**

- 同一本小说与同一个规范标签只能获得一次新增标签奖励。
- 标签别名与规范标签视为同一个标签。
- 标签奖励资格绑定“小说ID + 规范标签ID”，而不是绑定提交地址。
- 用户更换钱包地址重新提交，不产生新的奖励资格。
- 标签被删除后重新添加，原则上不得再次获得奖励。
- 只有原删除决定被正式推翻，且治理明确恢复奖励资格时，才可以例外处理。

建议记录：

`Rewarded[小说ID][规范标签ID]`

某个组合已经支付过有效奖励后，默认永久标记为已奖励。

**标签挑战**

任何人都可以按照规则质押代币，挑战错误的“小说—标签”关系。

有效挑战理由包括：

- 标签不符合其定义；
- 小说中不存在对应内容；
- 标签属于同义重复；
- 标签与已有标签存在明确冲突；
- 标签没有独立区分价值；
- 小说已经被认定为刷量小说；
- 提交者存在重复领取奖励行为；
- 提交者通过创建近义标签人为增加奖励；
- 存在其他违反标签规则的行为。

挑战成功后：

- 删除错误标签关系；
- 取消尚未支付的 `TagReward`；
- `TagDebit`退回标签虚拟余额；
- `BookDebit`退回小说标签助资余额；
- 不生成或撤销对应标签DAO声誉；
- 根据规则罚没提交者保证金；
- 挑战者可以获得清理奖励。

挑战失败后：

- 标签关系继续有效；
- 标签提交者获得原锁定的 `TagReward`；
- 挑战保证金按照规则部分罚没；
- 罚没资金可以补偿复核成本或进入标签DAO本级资金池。

不为普通“赞同已有标签”提供奖励，避免多个账户批量确认标签领取资金。

**刷量小说**

- 标签提交者默认担保该小说属于正常小说。
- 一本小说被认定为刷量小说后，禁止继续为其添加普通标签并领取奖励。
- 尚在挑战期的普通标签奖励全部取消。
- `TagDebit`退回对应标签虚拟余额。
- `BookDebit`退回对应小说标签助资余额。
- 已支付奖励是否追回，根据提交者当时是否能够合理识别刷量决定。
- 明知属于刷量小说仍批量添加标签的，可以撤销相关声誉并处罚保证金。
- “刷量”属于异常治理标签，不属于普通内容标签。

**标签合并**

同义、近义或没有独立语义的标签可以合并。

标签合并后：

- 选择一个规范标签；
- 其他标签变为规范标签的别名或废弃标签；
- 旧标签不再接受新的资金分配票；
- 旧标签不再产生新增奖励；
- 原资金分配票映射到规范标签；
- 原虚拟余额转入规范标签；
- 原小说标签关系迁移到规范标签；
- 重复关系只保留一条；
- 历史贡献记录继续保留。

规范标签合并后的余额为：

$$
NewCanonicalTagRemain
=
OldCanonicalTagRemain
+
MergedTagRemain

$$

同一用户同时资助多个被合并标签时：

- 资金分配票合并计算；
- 不因同义标签合并而增加总资金分配权。

如果一本小说同时拥有多个被合并标签：

- 合并后只保留一个规范标签关系；
- 不重复生成奖励；
- 不追溯追加奖励。

**标签废弃**

标签可以因为以下原因被废弃：

- 没有明确语义；
- 与现有标签完全重复；
- 定义无法稳定执行；
- 长期产生大量争议；
- 没有独立区分价值；
- 不适合作为小说标签；
- 已被新的规范标签替代。

标签废弃后：

- 不再接受资金分配票；
- 不再允许新增标签关系；
- 不再产生标签奖励；
- 历史标签关系和贡献记录继续保留；
- 剩余虚拟余额返回标签DAO本级，或者按照废弃提案转入替代标签。

标签合并和废弃只是标签DAO内部的虚拟账目调整，不属于子DAO撤销，不需要进行独立DAO清算。

## 标签DAO正式成员的职责

标签DAO正式成员主要处理：

- 新标签成立争议；
- 标签定义；
- 同义词和别名；
- 标签合并和废弃；
- 标签互斥关系；
- 错误标签挑战；
- 标签刷量事件；
- `RewardDivisor`调整；
- `R_max`调整；
- 标签虚拟余额异常；
- 小说标签助资使用规则；
- 标签奖励公式和参数更新。

普通标签提交仍采用乐观确认。

标签DAO重要事项使用专业票权：

`实际专业票权 = min（持币者有效专业委托票，活跃标签DAO声誉 × 声誉承载系数）`

标签DAO的重要治理决定仍保留全员复决窗口。

| ---------------------------------------------------------- | ---------- | ------------ | -------- |
| 无邀请的新用户                                           | 0%       | 0%         | 0%     |
| 有邀请的新用户                                           | 5%       | 5%         | 10%    |
| 成熟老用户                                               | 10%      | 0%         | 10%    |

| 邀请返佣必须在老用户优惠生效时终止否则会出现叠加漏洞：   |          |            |        |

1. A邀请B；
2. A长期获得B的返佣；
3. B之后成为老用户，又获得10%手续费优惠；
4. 同一个人同时获得老用户折扣和邀请返佣。

```

被邀请账户获得老用户优惠后，

```

方案

```

1. 满足老用户条件后，账户获得10%的长期手续费优惠。
2. 新用户通过邀请进入：

   - 新用户获得5%手续费优惠；
   - 邀请者获得5%手续费返佣。
3. 每笔手续费的协议总让利上限固定为10%。
4. 被邀请账户成为老用户后：

   - 自身优惠提高至10%；
   - 原邀请者返佣降为0。
5. 不设置额外的新用户注册奖励和固定邀请奖励。
6. 邀请者必须满足有效贡献等资格条件。
7. 返佣只基于实际支付并最终确认的协议净手续费。
```

# 治理

**总体原则**

- 每一级资金池都对应一个DAO，各级DAO按照树状结构组织。
- 各级DAO负责专业领域内的日常治理和资金使用，但所有决策的最终权力属于全体有效治理代币持有者。
- DAO不设置永久负责人。
- 日常事项采用默认通过的乐观治理。
- 重要专业事项由该级DAO的正式成员表决。
- 根本性事项直接由全体有效治理代币持有者表决。
- 任何下级DAO的重要决定，在达到复决门槛后，都可以提交全员复决。
- 每项具体任务、项目或提案必须有明确责任人，但责任人只对该事项负责，不形成永久治理职位。

DAO结构示例：
    主DAO
    ├─ 开发DAO
    │  ├─ 协议开发DAO：核心协议和智能合约
    │  ├─ 前端DAO：前端和用户体验
    │  ├─ 基础设施DAO：索引器、网关
    │  └─ 安全DAO：安全审计与漏洞奖励
    │  └─ 文档DAO
    ├─ 推广DAO
    │  ├─ 社区运营DAO：管理论坛等社区，解答问题
    │  ├─ 渠道推广DAO：通过各种传播渠道获取用户
    └─ 内容DAO
       ├─ 小说登记DAO
       ├─ 标签DAO
       ├─ 评分DAO
       ├─ 书单DAO
       └─ 刷量治理DAO

- 项目早期只需要实际运行主DAO、开发DAO、推广DAO和内容DAO。
- 更低级资金池可以先作为上级DAO内部的预算分类。
- 当某个预算分类拥有稳定任务、稳定资金和稳定贡献者后，再升级为独立DAO。
- 上级DAO确定下级DAO的职责边界和未来新增资金流向。
- 下级DAO决定已经进入其资金池的代币如何用于本领域任务。

**DAO不设置永久负责人**

- 乐观治理下，不需要为每个DAO设置永久负责人。
- 任何人都可以按照规定模板提交日常提案或任务提案。
- DAO没有永久负责人，不代表具体项目没有责任人。
- 每项提案或项目应明确：
  - 提案人；
  - 临时项目协调人；
  - 任务执行者；
  - 预算；
  - 劳动报酬；
  - 成本报销；
  - 交付物；
  - 完成期限；
  - 验收条件；
  - 付款条件；
  - 利益关系。
- 提案人负责回应挑战和补充信息。
- 项目协调人只负责某个具体项目，项目结束后不再拥有相关权限。
- 任务执行者负责具体交付物。
- 验收参与者负责在出现挑战时进行专业判断。
- 合约执行者只负责触发已经通过的链上操作，不拥有改变决策的权力。
- 路线图整理、文档维护、预算汇总等持续事务可以设置为周期性维护任务，由贡献者领取报酬。
- 周期维护者不是DAO负责人，也没有额外资金审批权。

**资金支出与自动执行**

- 资金分配票只决定未来新增代币进入哪个DAO，不直接授权提款。
- 日常支出应尽量通过提案、挑战期和托管合约自动执行。
- 普通任务流程可以为：
  `提交任务提案 → 公开挑战期 → 无挑战则自动生效 → 执行者完成任务 → 提交成果 → 验收挑战期 → 无挑战则自动付款并生成声誉`
- 提案通过后，资金可以进入项目托管合约。
- 达到预设付款条件后，任何人都可以调用执行函数。
- 调用者只能触发已经通过的付款，不能修改金额、收款方或用途。
- 如果暂时需要多签，多签只作为机械执行层。
- 多签不得自行改变治理结果。
- 小额任务可以在完成后一次性付款。
- 中型任务可以采用启动款和完成款。
- 大型任务应采用里程碑付款。
- 核心协议升级、安全审计、大额不可逆操作等高风险项目不能只依靠无人挑战自动验收，必须经过主动专业确认或投票。

**代币劳动价值**

- `1个代币 = 1个人1分钟的人类劳动价值`。
- 代币可以作为人类劳动报酬，也可以按照任务规则换算并补贴服务器、域名、网关、工具、代理、审计、第三方采购等其他开销。
- 声誉只对应已经确认的人类劳动时间，不用于补偿其他货币开销。
- 当需要换算成人民币参考值时，采用：
  `1个代币人民币参考值 = 中国国家统计局公布的最近一年城镇私营单位就业人员年平均工资 ÷ 250 ÷ 8 ÷ 60`
- 该人民币参考值只用于任务预算、劳动成本比较和费用核算。
- 它不是代币的市场价格、固定汇率、兑付价格或DAO回购承诺。
- 每次国家统计局公布新的年度数据后更新一次参考值。
- 已批准任务继续使用批准时的参考值。
- 新批准任务使用更新后的参考值。

**正式成员与专业人员**

- 达到某DAO规定声誉要求的贡献者，成为该DAO正式成员。
- 正式成员也即有资格接受该DAO专业治理委托的专业人员。
- 正式成员默认要求：
  - 累计声誉达到 `480点`；
  - 相关任务已经完成验收并通过挑战期；
  - 当前不存在未解决的严重违规。
- `480点`相当于一个标准工作日的劳动。
- 正式成员门槛不应过高，避免老成员垄断专业资格。
- 各DAO可以在 `120点至960点`范围内设置正式成员门槛。
- 不得按照现有成员平均声誉、中位数声誉、排名或历史最高值自动提高门槛。
- 达到公开客观条件后，应自动取得正式成员资格。
- 老成员不能无理由投票拒绝符合条件的新成员。
- 只有存在任务造假、抄袭、虚假验收、关联关系隐瞒等具体异议时，才进入审查程序。

正式成员资格本身不附带：

- 一人一票；
- 固定基础票；
- 按账户数量计算的票权；
- 按累计声誉直接增加的永久票权。

正式成员资格主要赋予：

- 接受专业治理委托的资格；
- 提交专业方案的资格；
- 参与专业讨论的资格；
- 参与评审和验收的资格；
- 申请成为临时项目协调人或周期维护者的资格；
- 在获得有效委托后参与重要专业提案表决的资格。

正式成员的实际专业票权为：
`实际专业票权 = min(持币者有效专业委托票, 活跃声誉 × 声誉承载系数)`

初始建议：
`1点活跃声誉最多承载10份专业委托票`

即：
`实际专业票权 = min(有效专业委托票, 活跃声誉 × 10)`

例如：

- 活跃声誉：`500点`
- 收到专业委托：`8,000票`
- 最大承载量：`5,000票`
- 实际专业票权：`5,000票`
- 超出的 `3,000票`暂时不产生治理权。

界面应向委托人显示：

- 专业人员当前活跃声誉；
- 最大承载量；
- 已收到委托；
- 实际生效委托；
- 超额无效委托。

持币者可以随时把超额委托转给其他专业人员。

该机制保证：

- 只有声誉、没有持币者委托，不产生实际专业票权；
- 只有大量委托、没有足够近期贡献，也不能行使全部票权；
- 正式成员身份本身不产生固定票；
- 创建大量正式成员小号不会因账户数量增加票权；
- 把同一批声誉和委托拆分到多个地址，不会增加合计票权。

不采用以下规则：

- 一人一票；
- 每个地址固定票；
- 按地址单独计算的平方根声誉票；
- 按地址单独计算的对数声誉票；
- 每个地址独立享有固定票权上限。

这些规则可能鼓励把任务、声誉和委托拆分到多个小号。

连续 `180天`没有在该DAO完成有效任务的正式成员，可以转为非活跃状态：

- 暂停评审、验收和临时协调资格；
- 不删除累计声誉；
- 重新完成有效任务后恢复活跃状态。

**三类委托**
以下三类委托相互独立：

- 资金分配委托：决定每日新增代币进入哪些DAO；
- 专业治理委托：决定某个DAO的重要专业提案；
- 全员治理委托：代表持币者参与主DAO级投票和最终复决。
- 持币者可以把三类委托交给不同的人。
- 持币者可以在具体投票中覆盖代理人的选择。
- 持币者可以随时撤销或更换委托。
- 早期不允许专业人员再次转委托，避免形成不透明的多层委托链。

**日常治理**

日常事项采用默认通过的乐观治理机制。

适用事项包括：

- 发布普通任务；
- 在既定规则和预算范围内设置悬赏；
- 小额支出；
- 常规维护；
- 普通内容规则执行；
- 已批准项目的普通里程碑结算；
- 不改变上级授权范围的内部安排；
- 按公开公式计算的标准奖励。

流程为：
`公开提案 → 挑战期 → 无有效挑战 → 自动通过并执行`

挑战期可以设置为：

- 普通日常事项：`3天`
- 涉及付款或关联方：`5至7天`
- 高风险但仍可乐观执行的事项：根据规则适当延长

以下主体可以提出挑战：

- 任意有效治理代币持有者；
- 对应DAO的正式成员；
- 受到该事项直接影响的人。

普通治理挑战应：

- 指出具体问题；
- 说明违反的规则；
- 提供证据或合理风险说明；
- 按规定质押挑战保证金。

无人挑战时不进行投票。

日常事项被有效挑战后：
`日常提案 → 有效挑战 → 该级DAO正式成员专业表决`

专业表决采用：
`实际专业票权 = min(持币者有效专业委托票, 活跃声誉 × 声誉承载系数)`

**重要专业事项**

以下事项通常由该级DAO正式成员直接表决，不采用默认通过：

- 较大金额项目；
- 高风险项目；
- 修改本DAO内部支出规则；
- 创建或显著调整下级DAO；
- 选择重要项目方案；
- 处理复杂专业争议；
- 正式成员处罚；
- 被挑战日常提案的专业裁决；
- 超出普通乐观治理范围的事项。

该级DAO表决通过后，不立即成为不可推翻的最终决定，而是进入全员复决窗口。

**全员最终权力**

- 所有DAO决策的最终权力属于全体有效治理代币持有者。
- 各级DAO正式成员拥有专业判断和日常治理权，但不拥有不可推翻的最终权力。
- 任何下级DAO的重要决定通过后，都应保留全员复决窗口。
- 达到全员复决触发门槛后，原决定暂停执行并提交全员投票。
- 全员投票可以：
  - 维持原决定；
  - 撤销原决定；
  - 要求修改后重新提交；
  - 退回相应DAO重新处理。

权力顺序为：

`提案执行者 < 该级DAO专业治理 < 上级DAO < 全体有效治理代币持有者`

全员复决触发门槛应按有效治理代币计算，不按地址数量计算。

项目早期可以采用以下任一条件触发全员复决：

- 获得全体有效治理票的 `0.05%至0.1%`支持；
- 获得该级DAO有效专业委托票的一定比例支持。

全员投票按照有效治理代币加权。

以下代币不参与全员投票：

- 各级DAO国库持有的代币；
- 协议收入国库持有的代币；
- 发行合约持有的代币；
- 项目托管合约持有的代币；
- 其他协议控制地址持有的代币。

全员投票结果对本次事项具有终局性。

再次提交相同事项必须满足至少一个条件：

- 出现新的事实或证据；
- 提案有实质性修改；
- 已经过规定的冷却期。

**必须直接全员投票的事项**

以下事项不能只由专业DAO决定，必须直接提交全员投票：

- 修改每日增发上下限或增发公式；
- 修改资金分配规则；
- 修改专业委托规则；
- 修改声誉生成规则；
- 修改正式成员门槛范围；
- 修改活跃声誉衰减规则；
- 修改声誉承载系数；
- 创建、合并或永久解散一级DAO；
- 修改主DAO与一级DAO之间的权力结构；
- 升级核心协议合约；
- 修改全员复决权；
- 动用重大比例的主DAO国库；
- 永久冻结资金或权限；
- 修改协议宪法性规则。

专业DAO负责形成可执行方案，全员负责决定是否最终授权。

**紧急安全暂停**

紧急安全事件使用独立于普通治理挑战的快速通道。

任何人都可以：
`提交安全报告 + 质押代币 → 立即暂停最小必要范围`

不要求报告人是正式成员，也不要求提前获得批准。

安全报告至少应说明：

- 受影响的DAO、资金池或合约；
- 风险类型；
- 可能损失的资金或权限；
- 可验证的证据或复现方法；
- 建议暂停的范围。

可暂停的对象包括：

- 某个项目托管合约；
- 某个下级DAO资金池；
- 某个一级DAO；
- 某个核心协议模块；
- 整个协议。

暂停必须遵循最小必要范围原则。

局部风险不得默认暂停整个协议。

暂停期间通常：

- 禁止受影响资金转出；
- 禁止执行相关待处理提案；
- 禁止修改受影响权限；
- 允许资金转入；
- 允许提交证据、审计和修复方案。

暂停保证金可以采用：
`暂停保证金 = max(最低保证金, 预计受威胁资金 × 保证金比例)`

同时设置最高保证金，避免大型资金池因门槛过高而无人能够报告。

首次暂停立即生效，但持续时间有限：

- 普通资金池：最长约 `24小时`
- 一级DAO或核心模块：最长约 `12小时`
- 全协议暂停：最长约 `6小时`

范围越大，未经确认的自动暂停时间越短。

暂停后，由受影响DAO中无利益冲突的正式成员快速复核，可以作出：

- 确认存在紧急风险；
- 确认风险但缩小暂停范围；
- 证据不足并解除暂停；
- 认定为恶意或明显滥用。

专业复核只能暂时延长暂停和启动修复，不能永久冻结资金。

建议期限：

- 首次自动暂停：`6至24小时`
- 专业确认后：最多延长至 `7天`
- 超过 `7天`：必须由全员投票批准
- 全员每次批准延长：最长 `30天`

以下事项必须提交全员最终确认：

- 暂停一级DAO；
- 暂停整个协议；
- 暂停超过规定期限；
- 升级核心合约；
- 永久废弃某个合约；
- 永久冻结资金或权限；
- 专业DAO对安全事件存在重大分歧。

**安全奖励与保证金处理**

安全事件被确认后：
`报告人保证金全额退还 + 获得受影响资金池支付的代币奖励`

安全奖励包括：
`安全奖励 = 基础劳动奖励 + 风险奖励`

基础劳动奖励按照调查、复现和报告所需的合理劳动时间计算：
`基础劳动奖励 = 经确认的有效劳动分钟数 × 1代币`

风险奖励可以采用：
`风险奖励 = min(确认受威胁资金 × 奖励比例, 受影响资金池余额 × 奖励上限比例)`

例如可以初始设置：

- 受威胁资金奖励比例：`1%`
- 单次奖励最多不超过受影响资金池余额的 `5%`

奖励来源：

- 下级DAO安全事件由该下级DAO资金池支付；
- 下级DAO余额不足时，由其直接上级DAO补足；
- 一级DAO余额不足时，由主DAO安全储备补足；
- 不为安全奖励额外增发代币。

多人报告同一事件时：

- 第一个提供足够证据并使风险得到确认的人获得主要奖励；
- 后续提供新的关键证据或修复方案的人可以获得补充奖励；
- 单纯重复报告不重复奖励；
- 极短时间内独立提交的有效报告，可以按证据贡献分配奖励。

报告未被确认时：

- 善意但证据不足：退还大部分保证金，仅扣除复核成本；
- 明显错误或严重草率：部分罚没保证金；
- 伪造证据、恶意暂停或重复骚扰：全部罚没保证金，并限制再次触发；
- 被罚没的保证金进入受影响DAO资金池。

以下人员原则上不得获得风险奖励：

- 故意植入漏洞的人；
- 与漏洞制造者串通的人；
- 通过自己控制的权限人为制造风险的人；
- 先利用漏洞获利再提交报告的人；
- 隐瞒已经造成的损失或攻击行为的人。

开发者无意引入漏洞后主动报告的，可以获得合理的调查和修复劳动报酬；是否获得风险奖励，根据过失程度另行判断。

安全报告可以分为：

- 公开部分：受影响范围、风险等级、暂停对象、报告时间；
- 保密部分：具体攻击路径、代码位置、密钥或复现细节。

技术细节可以在修复完成前暂时保密，但以下信息必须公开：

- 暂停范围；
- 暂停期限；
- 确认结果；
- 保证金；
- 最终奖励；
- 最终处理决定。

所有紧急暂停和安全处理仍然服从全体有效治理代币持有者的最终复决权。

**核心规则汇总**

1. 每一级资金池对应一个DAO。
2. DAO采用树状结构，上级DAO管理职责边界和未来资金流向，下级DAO管理自身预算的具体执行。
3. DAO不设置永久负责人，每个具体提案、任务和项目设置临时责任人。
4. 所有每日新增代币按照持币者直接分配票和专业委托票递归进入各级DAO。
5. 持币者可以随时修改资金分配和委托，修改从下一个每日周期开始生效。
6. 已经进入DAO的资金不会因后续改票而移动。
7. 已有资金不设置长期闲置自动回收机制。
8. DAO国库和协议控制地址持有的代币不产生资金分配权和全员治理权。
9. `1个代币`代表`1个人1分钟`的人类劳动价值，也可以按任务规则换算并补贴其他货币开销。
10. `1分钟`已经确认的人类劳动时间，获得对应DAO的1点不可转让声誉。
11. 成本报销、采购费用、保证金返还、服务器费用、工具费用等非人类劳动支出不产生声誉。
12. 正式成员资格由该DAO的累计声誉和有效任务数量判断。
13. 正式成员门槛应保持较低，默认可以采用480点声誉和2个独立任务。
14. 正式成员身份本身不附带一人一票或固定票权。
15. 重要专业提案的票权由持币者专业委托和活跃声誉共同限制。
16. 日常决策采用默认通过的乐观治理，无挑战时不投票。
17. 日常任务完成后的普通验收也可以采用默认通过机制。
18. 重要专业事项由该级DAO获得有效委托的正式成员表决。
19. 根本性事项直接由全体有效治理代币持有者表决。
20. 所有下级DAO的重要决定都可以在达到门槛后提交全员复决。
21. 任何人都可以通过质押代币立即触发最小必要范围的紧急安全暂停。
22. 安全事件确认后，报告人获得保证金退还和相应资金池支付的奖励。
23. 日常执行和专业判断可以下放，但协议最终权力始终属于全体有效治理代币持有者。

## 预算分类

**预算分类的定位**

- 预算分类是正式DAO内部用于组织任务、限制支出和统计成果的会计分类。
- 预算分类不是独立DAO，也不是独立资金池。
- 预算分类只有用途范围和周期支出额度，不拥有独立资产、成员资格或治理权。
- 预算分类主要解决以下问题：
  - 将不同类型的任务分开管理；
  - 限制某类任务在一个预算周期内最多支出多少；
  - 统计不同工作领域的投入和成果；
  - 判断某个领域以后是否需要升级为独立子DAO。

基本原则：
`预算分类 = 任务标签 + 周期支出上限 + 独立统计`

而不是：
`预算分类 = 独立资金池或子DAO`

例如开发DAO内部可以设置：

- 协议开发；
- 前端；
- 基础设施；
- 安全；
- 开发工具与文档。

这些分类都属于开发DAO，不是独立DAO。

**预算分类不具备的权利**

预算分类不具备：

- 独立资金池；
- 独立国库地址；
- 独立DAO合约；
- 独立正式成员资格；
- 独立专业治理委托；
- 独立资金分配票；
- 独立每日增发入口；
- 独立提案表决权；
- 独立资金转账权；
- 永久积累预算余额的权利。

如果一个分类开始：

- 独立持有资金；
- 接受持币者直接分配票；
- 建立自己的正式成员；
- 建立独立声誉；
- 独立决定项目和支出；

则它已经接近子DAO，应当进入试运行子DAO或正式子DAO流程，不应继续作为普通预算分类。

**资金归属**

- 所有资金始终保存在预算分类所属的正式DAO资金池中。
- 预算分类只记录该类别可以使用的最高额度。
- 将额度分配给预算分类，不代表已经把代币转入另一个账户。

例如：

- 开发DAO资金池总余额：`100,000代币`
- 本周期可安排预算：`60,000代币`

本周期预算分类可以设置为：

- 协议开发：最高 `24,000代币`
- 前端：最高 `15,000代币`
- 基础设施：最高 `9,000代币`
- 安全：最高 `9,000代币`
- 开发工具与文档：最高 `3,000代币`

这只是支出额度安排。

开发DAO资金仍统一保存在开发DAO资金池中，不会实际转入五个独立分类账户。

**预算周期**

- 每个正式DAO可以设置固定预算周期。
- 预算周期可以与每日增发周期不同。
- 建议早期采用 `30天` 或 `90天`作为一个预算周期。
- 每个预算周期开始前或开始后，应公开新的分类预算安排。
- 分类预算只对当前预算周期有效。
- 未使用的分类额度不会自动延续为该分类的永久余额。
- 新预算周期开始时，应根据DAO余额、任务需求和过去支出重新制定额度。

预算周期结束时：

- 已经支付的金额计入实际支出；
- 已经批准但尚未完成的项目计入已承诺金额；
- 尚未使用且没有项目承诺的额度自动失效；
- 失效的只是分类额度，不是DAO资金；
- 未使用资金继续保留在上级DAO资金池中。

**分类预算的制定**

每个预算周期可以提交一个整体预算包。

预算包应说明：

- DAO当前总余额；
- 已经承诺给现有项目的金额；
- 本周期预计新增资金；
- 本周期预计可支出总额；
- 各预算分类的额度；
- 本级机动额度；
- 各分类的主要任务目标；
- 与上一个周期相比发生的主要变化。

例如：

- 协议开发：`35%`
- 前端：`25%`
- 基础设施：`15%`
- 安全：`15%`
- 开发工具与文档：`5%`
- 本级机动额度：`5%`

各分类额度与机动额度之和不能超过本周期可安排预算。

预算额度可以按比例表示，也可以按具体代币数量表示。

如果DAO余额和每日流入变化较大，可以同时规定：

`分类预算上限 = min（固定代币额度，本周期实际可用资金 × 分类比例）`

**预算包的治理方式**

普通预算包可以采用乐观治理：

`公开预算包 → 进入挑战期 → 无有效挑战 → 自动生效`

适合乐观通过的情况包括：

- 各分类职责没有变化；
- 预算结构与上一周期相比变化较小；
- 不涉及异常大额支出；
- 不涉及明显利益冲突；
- 不改变DAO的基本治理规则。

出现以下情况时，应由该级DAO正式成员进行专业表决：

- 某个分类占本周期预算比例过高；
- 某个分类额度与上一周期相比大幅增加；
- 大幅削减其他重要分类预算；
- 涉及核心协议、安全或重大不可逆操作；
- 预算方案存在明显利益冲突；
- 预算超过乐观治理允许的额度范围；
- 预算包受到有效挑战。

该级DAO专业表决通过后，仍保留全员复决窗口。

**建立预算分类**

任何人都可以在正式DAO内部提出建立新的预算分类。

新增分类提案应说明：

- 分类名称；
- 所属DAO；
- 职责范围；
- 允许包含的任务类型；
- 不允许包含的任务类型；
- 与现有分类的区别；
- 为什么不能归入现有分类；
- 初始预算上限；
- 试用期限或复查日期；
- 预期可以改善的问题。

普通预算分类可以采用乐观成立：
`提交分类提案 → 公开挑战期 → 无有效挑战 → 自动建立`

如果受到有效挑战，则由所属DAO正式成员专业表决。

预算分类的建立不会自动获得预算。

分类建立后，仍需通过本周期预算包或预算调整提案为其设置额度。

**预算分类的职责边界**

每个预算分类应明确职责边界，避免同一任务可以随意选择多个分类。

例如开发DAO可以规定：

- 协议开发：

  - 智能合约；
  - 核心业务逻辑；
  - 协议接口；
  - 协议升级。
- 前端：

  - 网页界面；
  - 移动端适配；
  - 钱包交互；
  - 用户体验。
- 基础设施：

  - 索引器；
  - 网关；
  - 节点；
  - 数据同步；
  - 部署和监控。
- 安全：

  - 安全审计；
  - 漏洞修复；
  - 漏洞奖励；
  - 权限检查；
  - 应急响应。
- 开发工具与文档：

  - SDK；
  - 测试工具；
  - 开发文档；
  - 接口文档；
  - 示例项目。

分类边界属于预算和统计规则，不代表相关任务只能由某类固定成员执行。

**任务占用预算**

每个任务提案必须选择所属预算分类。

任务提案应至少包含：

- 所属DAO；
- 所属预算分类；
- 任务目标；
- 劳动报酬；
- 成本报销；
- 交付物；
- 完成期限；
- 验收条件；
- 收款地址；
- 关联关系。

例如：

- 所属DAO：开发DAO
- 预算分类：前端
- 任务：改进移动端书籍详情页
- 劳动报酬：`1,200代币`
- 成本报销：`0`
- 交付物：页面代码、测试结果和部署说明

任务进入治理流程前，系统应检查：

1. 任务是否属于该分类职责；
2. 分类剩余可安排额度是否足够；
3. 任务金额是否符合乐观治理上限；
4. 是否存在利益冲突；
5. 是否需要主动专业验收。

任务通过后：

- DAO资金仍保存在正式DAO资金池中；
- 该分类的已承诺金额增加；
- 分类剩余可安排额度减少；
- 项目资金可以进入项目托管合约；
- 项目完成并验收后，由正式DAO资金池或托管合约付款。

预算分类的基本记账关系为：
`剩余可安排额度 = 分类预算上限 - 已承诺金额`

实际支出完成后：
`分类实际支出 = 已完成付款的任务金额`

**已承诺金额与实际支出**

预算分类应分别记录：

- 预算上限；
- 已承诺金额；
- 已支付金额；
- 已取消项目释放的金额；
- 剩余可安排额度。

例如：

- 前端分类预算上限：`25,000代币`
- 已承诺项目：`8,000代币`
- 已实际支付：`5,000代币`
- 剩余可安排额度：`17,000代币`

其中尚未支付的 `3,000代币`仍属于已承诺资金，不能再次用于其他任务。

项目被取消后，尚未支付的承诺金额重新释放回该分类的可安排额度。

**分类额度不是必须花完**

- 分类预算是最高可支出额度，不是必须完成的支出指标。
- 不要求分类在周期结束前把预算全部用完。
- 不得为了避免预算失效而发布无实际价值的任务。
- 未使用额度自动失效，资金继续留在正式DAO资金池。
- 下一个周期根据实际需求重新分配。

例如：

- 前端分类预算上限：`25,000代币`
- 本周期实际支出：`12,000代币`
- 已承诺但未支付：`3,000代币`
- 未使用额度：`10,000代币`

周期结束后：

- `12,000代币`计入实际支出；
- `3,000代币`继续作为项目承诺；
- 未使用的 `10,000代币`额度失效；
- 对应资金仍留在开发DAO资金池。

**跨分类项目**

一个项目可以同时涉及多个预算分类。

跨分类项目应在提案时明确：

- 主分类；
- 参与分类；
- 每个分类承担的预算；
- 各分类对应的交付物；
- 项目协调方式；
- 验收责任。

例如搜索功能升级项目：

- 协议开发：`2,000代币`
- 前端：`1,500代币`
- 基础设施：`500代币`

可以采用两种方式：

1. 拆分成多个独立子任务，分别占用各分类额度；
2. 保留一个联合项目，但明确每个分类承担的金额。

不能在项目付款时临时决定从哪个分类扣除预算。

**本级机动额度**

每个正式DAO可以保留一定比例的本级机动额度。

机动额度可用于：

- 尚未建立预算分类的新任务；
- 跨分类项目；
- 某分类的小额预算不足；
- 无法明确归类的公共任务；
- 紧急但不属于安全暂停的支出。

建议机动额度占本周期可安排预算的 `5%至10%`。

机动额度不能用于：

- 长期绕过分类预算；
- 隐藏某类任务的真实支出；
- 为关联方提供未经披露的资金；
- 拆分大额项目以逃避专业表决。

超过规定金额的机动额度支出，仍需该级DAO专业表决。

**预算调整**

预算周期内可以提出调整分类额度。

调整只改变正式DAO内部的支出授权，不发生真实资金转账。

例如：

- 前端分类增加：`5,000代币`
- 开发工具与文档分类减少：`5,000代币`

开发DAO资金池总余额不发生变化。

小幅预算调整可以采用乐观治理：
`提交调整提案 → 挑战期 → 无挑战 → 自动生效`

较大调整应由该级DAO正式成员专业表决，例如：

- 调整金额超过本周期可安排预算的一定比例；
- 大幅削减安全等重要分类；
- 将大量机动额度转入单一分类；
- 涉及关联项目；
- 调整受到有效挑战。

调整预算时，不能取消已经通过且仍然有效的项目承诺。

**预算不足**

某个分类预算不足时，可以：

- 等待下一个预算周期；
- 提交预算调整提案；
- 从本级机动额度申请；
- 降低任务预算；
- 分阶段实施任务；
- 重新安排任务优先级；
- 将确实涉及其他领域的部分划入其他分类。

不能在分类预算不足时直接绕过分类规则付款。

**预算分类与声誉**

预算分类阶段不建立独立的正式成员资格和独立治理声誉。

在某预算分类完成任务后：

`获得1代币有效劳动报酬 → 获得所属正式DAO的1点声誉`

例如在开发DAO的前端分类完成任务并获得 `1,000代币`劳动报酬：

- 获得开发DAO声誉：`1,000点`
- 记录前端分类贡献：`1,000点`

前端分类贡献记录用于：

- 展示贡献者的专业经历；
- 统计该领域的有效劳动投入；
- 判断是否形成稳定的独立工作领域；
- 判断是否需要升级为试运行子DAO；
- 为未来子DAO声誉迁移提供依据。

前端分类贡献记录在预算分类阶段：

- 不产生独立正式成员资格；
- 不接受独立专业治理委托；
- 不直接产生独立投票权。

**升级为试运行子DAO**

预算分类出现以下情况时，可以考虑升级为试运行子DAO：

- 长期存在持续任务；
- 职责已经相对独立；
- 有必要接受持币者直接资金分配票；
- 有必要建立独立声誉和正式成员资格；
- 有必要独立管理项目和资金；
- 上级DAO统一治理已经明显降低效率；
- 分类的工作规模足以承担独立治理成本。

升级提案应说明：

- 为什么预算分类已经不足；
- 未来子DAO的职责；
- 试运行期限；
- 试运行预算；
- 历史分类贡献如何映射；
- 与上级DAO和其他分类的边界；
- 试运行失败后的处理方式。

升级为试运行子DAO后：

- 原预算分类停止接受新任务；
- 未完成任务继续由原正式DAO负责，或者按升级提案移交；
- 原分类未使用额度失效；
- 资金继续保留在原正式DAO，除非试运行预算提案另有安排；
- 符合条件的历史分类贡献可以映射为试运行子DAO声誉。

历史贡献映射应满足：

- 任务执行前已经明确标记为该预算分类；
- 任务真实完成并通过挑战期；
- 只有劳动报酬部分可以映射；
- 同一劳动贡献不能重复生成声誉；
- 映射规则必须在试运行子DAO成立前公开。

例如：

- 历史前端分类有效劳动报酬：`800代币`
- 开发DAO累计声誉继续保留：`800点`
- 试运行前端DAO记录前端专业声誉：`800点`

这是同一贡献在上级领域和具体子领域中的分层记录，不是重复发放代币或重复增加同一级治理票权。

**预算分类的修改**

预算分类可以修改：

- 名称；
- 职责范围；
- 允许的任务类型；
- 与其他分类的边界；
- 预算统计方式；
- 复查周期。

普通修改采用乐观治理：
`提交修改提案 → 挑战期 → 无挑战 → 自动生效`

以下修改应由该级DAO正式成员专业表决：

- 大幅改变分类职责；
- 将多个重要分类合并；
- 把高风险事项转入普通分类；
- 修改可能影响大量现有项目；
- 修改受到有效挑战。

修改分类不得追溯改变已经批准项目的所属分类和预算规则，除非相关项目提案同步获得修改批准。

**预算分类的撤销**

预算分类可以在以下情况下撤销：

- 长期没有相关任务；
- 职责已经并入其他分类；
- 与其他分类高度重叠；
- 分类边界不再合理；
- 已经升级为试运行或正式子DAO；
- 上级DAO业务结构发生变化。

普通撤销可以采用乐观治理：
`提交撤销提案 → 挑战期 → 无挑战 → 自动撤销`

预算分类撤销时：

- 停止接受新的任务；
- 未使用预算额度自动失效；
- 资金继续保留在所属正式DAO资金池；
- 已完成任务和支出记录永久保留；
- 已获得的正式DAO声誉继续保留；
- 分类贡献记录继续作为历史记录保留；
- 已经批准的项目不能直接取消；
- 未完成项目应继续完成、终止或重新归入其他分类；
- 已承诺资金按照项目处理结果释放或支付。

预算分类撤销不需要正式资产清算，因为它不独立持有资金。

**预算分类与子DAO的区别**


| 项目                     | 预算分类                  | 试运行子DAO          | 正式子DAO          |
| -------------------------- | --------------------------- | ---------------------- | -------------------- |
| 独立资金池               | 无                        | 原则上无，可独立记账 | 有                 |
| 独立持有资金             | 否                        | 有限或由上级托管     | 是                 |
| 接收每日新增代币         | 否                        | 默认否               | 是                 |
| 接受持币者直接资金分配票 | 否                        | 默认否               | 是                 |
| 独立声誉                 | 无，只记录分类贡献        | 有试运行声誉         | 有正式声誉         |
| 独立正式成员             | 无                        | 可积累资格           | 有                 |
| 专业治理委托             | 无                        | 可以有限测试         | 有                 |
| 独立提案流程             | 无                        | 有限                 | 有                 |
| 未使用预算处理           | 额度失效，资金留在上级DAO | 按试运行规则处理     | 资金留在自身资金池 |
| 撤销是否需要清算         | 不需要                    | 简化处理             | 需要正式清算       |
| 主要作用                 | 组织任务和限制支出        | 验证独立治理必要性   | 长期独立治理       |

## 子DAO的成立与撤销

**子DAO生命周期**
子DAO按照以下路径演进：
`预算分类 → 试运行子DAO → 正式子DAO → 正常运行 → 休眠、合并或撤销`

- 不预先建立大量正式子DAO。
- 一个专业领域最初只是上级DAO内部的预算分类。
- 只有形成持续任务、独立职责和实际治理需求后，才逐步升级为正式子DAO。
- 子DAO的成立、休眠、合并和撤销，最终均服从全体有效治理代币持有者的复决权。

**预算分类阶段**

一个新的工作领域最初作为上级DAO内部的预算分类存在。

例如：
    开发DAO
    ├─ 协议开发预算
    ├─ 前端预算
    ├─ 基础设施预算
    └─ 安全预算

预算分类阶段具有以下特点：

- 没有独立资金池；
- 没有独立DAO合约；
- 没有独立正式成员资格；
- 没有独立专业治理委托；
- 没有独立声誉体系；
- 不接受持币者直接分配资金票；
- 任务、资金、声誉和治理仍然归直接上级DAO管理。

当该领域逐渐形成独立且持续的工作需求后，任何人都可以提出建立试运行子DAO。

**试运行子DAO的成立**

任何人都可以提出试运行子DAO成立提案，不要求提案人已经是正式成员。
成立提案必须明确：

- 子DAO名称；
- 直接上级DAO；
- 职责范围；
- 与现有DAO及预算分类的区别；
- 预计处理的任务类型；
- 试运行期限；
- 试运行预算上限；
- 资金来源；
- 任务验收方式；
- 声誉生成规则；
- 正式成立条件；
- 试运行失败后的资金、任务和声誉处理方式。

试运行期限建议为一个完整预算周期，可以设置为 `30至90天`。

试运行子DAO可以采用乐观成立机制：
`提交完整提案 → 进入挑战期 → 无有效挑战 → 自动开始试运行`

- 试运行成立提案应公开不少于规定期限。
- 无有效挑战时，不需要进行正式投票。
- 出现有效挑战时，由直接上级DAO的正式成员进行专业表决。
- 专业表决通过后，仍保留全员复决窗口。
- 达到全员复决触发门槛后，由全体有效治理代币持有者作出最终决定。

**试运行阶段的权限**
试运行子DAO可以：

- 建立独立的提案列表；
- 发布本领域任务；
- 记录本领域任务成果；
- 生成该试运行子DAO的声誉；
- 设置临时项目托管账户或独立记账账户；
- 测试适合本领域的任务、验收和治理规则。

试运行子DAO暂时不能：

- 自动获得每日新增代币；
- 自动获得固定预算比例；
- 永久控制独立资金池；
- 建立永久特殊成员；
- 创建永久管理员或负责人；
- 擅自改变上级DAO的职责和资金规则。

试运行资金由上级DAO通过单独提案批准，可以采用：

- 上级DAO直接托管；
- 独立子账户记账；
- 项目托管合约；
- 设置明确上限的试运行预算。

试运行子DAO获得的预算，不代表其以后自动拥有相同比例的长期资金。

**试运行期间的声誉**
试运行期间完成的任务可以生成该子DAO声誉。

规则为：
`1分钟已经确认的人类劳动时间 = 获得该试运行子DAO的1点声誉`

- 只有已经确认的人类劳动时间产生声誉。
- 成本报销、采购费用、保证金返还等非劳动款项不产生声誉。
- 试运行任务产生的声誉同时按规则计入直接上级DAO。
- 试运行子DAO正式成立后，试运行期间获得的有效声誉继续保留。
- 如果试运行失败，相关声誉仍作为真实历史贡献保留，但不再产生一个不存在DAO的专业治理权。
- 若任务被确认造假、抄袭或无效，则撤销该任务产生的声誉。

**正式成立条件**
子DAO正式成立主要根据实际任务和治理需求判断，不以地址数量或成员人数作为主要条件。

建议至少满足：

- 已完成一个完整试运行周期；
- 已完成至少 `2个`实质独立任务；
- 试运行期间累计确认的有效劳动报酬达到一定规模，例如不少于 `960代币`；
- 已产生可验证的实际成果；
- 该领域具有持续工作需求，而不是一次性项目；
- 职责能够与上级DAO及其他子DAO清楚区分；
- 独立记账和治理能够提高效率、透明度或专业性；
- 至少有一名达到该子DAO正式成员声誉要求的贡献者；
- 不存在尚未解决的重大资金、任务或声誉争议；
- 已经制定正式运行后的职责、治理、资金和解散规则。

不得把以下条件作为主要成立依据：

- 钱包地址数量；
- 群聊人数；
- 社区注册人数；
- 支持账户数量；
- 容易通过小号制造的成员数量。

**正式成立程序**

二级及更低级子DAO正式成立流程为：
`提交试运行报告 → 直接上级DAO正式成员专业表决 → 进入全员复决窗口 → 未触发复决或全员维持决定 → 正式成立`

专业成员的实际票权为：
`实际专业票权 = min（持币者有效专业委托票，活跃声誉 × 声誉承载系数）`

- 正式成员不采用一人一票。
- 正式成员身份本身不产生固定票权。
- 子DAO正式成立属于重要专业事项，不能只通过普通日常乐观提案完成。
- 全体有效治理代币持有者保留最终复决权。

新增一级DAO，与开发DAO、推广DAO和内容DAO并列时，必须直接由全员投票决定，不能只由某个上级专业DAO批准。

**正式成立后的权限**

正式子DAO可以获得：

- 独立资金池或独立国库子账户；
- 独立声誉记录；
- 独立正式成员资格；
- 独立专业治理委托；
- 独立提案和挑战流程；
- 接受持币者直接资金分配票的资格；
- 在职责范围内发布任务和管理项目的权限；
- 建立本级预算分类的权限；
- 提出成立更低级子DAO的权限。

正式成立不代表自动获得资金。

子DAO正式成立后：
`无人分配资金票 → 每日新增资金为0`

子DAO只有在持币者或受委托专业人员主动将上级DAO中的票继续分配给它后，才获得未来每日新增代币。

没有继续向下分配的票，对应资金仍然留在直接上级DAO本级。

上级DAO可以为新成立子DAO批准一次有限的启动预算，但不能自动赋予其永久固定拨款比例。

**创立者和正式成员**

- 子DAO创立提案人不会因为提出成立方案而自动成为正式成员。
- 子DAO首批参与者不会自动获得永久特殊权力。
- 正式成员仍需达到统一的客观声誉条件。
- 正式成员资格不因“创始成员”身份永久保留。
- 子DAO不得向创立者授予永久否决权、永久负责人职位或固定资金比例。

正式成员可以按照统一规则认定，例如：

- 累计声誉达到 `480点`；
- 至少完成 `2个`独立任务；
- 任务已经完成验收并通过挑战期；
- 当前不存在未解决的严重违规。

**子DAO休眠**
子DAO暂时没有足够工作，但未来可能恢复时，应优先进入休眠，而不是立即永久撤销。

可以提出休眠的情况包括：

- 长期没有新的有效任务；
- 当前阶段性目标已经完成；
- 暂时没有符合条件的活跃专业成员；
- 工作量不足以维持独立治理；
- 正在准备与其他DAO合并；
- 当前资金和任务适合由上级DAO暂时代管。

休眠属于重要事项，由直接上级DAO正式成员专业表决，并保留全员复决窗口。

进入休眠后：

- 暂停接受新的普通任务提案；
- 暂停新的日常支出；
- 不得建立新的长期项目承诺；
- 已批准项目可以继续完成；
- 已确认债务和劳动报酬应继续支付；
- 已有资金不自动回收；
- 已有声誉和历史记录继续保留；
- 已有正式成员身份记录继续保留；
- 暂停新的专业治理活动。

休眠提案必须明确是否暂停接收未来每日新增资金。

推荐默认规则为：
`进入休眠 → 暂停接收未来新增资金 → 原本对应的新增资金留在直接上级DAO本级`

- 这只停止未来资金流入，不回收子DAO已有资金。
- 持币者原有分配配置可以保留，以便子DAO恢复时继续生效。
- 也可以要求系统提示持币者重新调整分配。

**子DAO重新激活**

任何人都可以提出重新激活休眠子DAO。

重新激活提案必须说明：

- 新出现的持续任务需求；
- 预计任务和预算；
- 可执行的工作计划；
- 当前专业贡献能力；
- 重新激活后的目标；
- 试运行或观察期限；
- 现有规则是否需要修改。

推荐流程为：
`提交重新激活提案 → 挑战期 → 无挑战 → 恢复为短期试运行状态`

- 完成新的短期试运行后，再恢复正式活跃状态。
- 如果休眠时间较短，职责、规则和成员结构没有明显变化，也可以由上级DAO专业表决直接恢复正式运行。
- 重新激活后，是否恢复接收每日新增资金，应在提案中明确。
- 原有资金和历史声誉继续保留。

**子DAO合并**

两个或多个子DAO存在以下情况时，可以提出合并：

- 职责高度重叠；
- 长期共同处理同一类项目；
- 独立治理成本明显高于专业差异带来的收益；
- 一个DAO长期依赖另一个DAO；
- 各自工作量不足以维持独立治理；
- 协议架构调整后，原有划分不再合理。

合并提案必须说明：

- 参与合并的DAO；
- 合并后的DAO名称；
- 合并后的职责范围；
- 新DAO的直接上级；
- 原资金如何迁移；
- 未完成项目如何处理；
- 债务和待付款项如何处理；
- 历史声誉如何映射；
- 原正式成员资格如何处理；
- 原DAO权限何时关闭；
- 历史记录如何归档。

同一上级DAO下的子DAO合并流程为：
`提交合并提案 → 直接上级DAO正式成员专业表决 → 全员复决窗口 → 执行合并`

不同一级DAO之间的合并，或者一级DAO本身的合并，必须直接由全员投票决定。

资金处理原则：
`原DAO剩余资金 → 转入合并后的DAO资金池`

该资金迁移属于经过治理批准的组织合并，不属于长期闲置资金自动回收。

声誉处理原则：

- 原DAO历史声誉永久保留；
- 与新DAO职责一致的声誉可以映射为新DAO声誉；
- 同一份声誉不得在合并后重复计算；
- 已经计入上级DAO的声誉不重复增加；
- 与新DAO职责不相关的声誉只作为历史记录保留；
- 合并规则必须在提案通过前公开。

**子DAO永久撤销**

永久撤销适用于：

- 该领域任务已经彻底完成；
- 职责被其他DAO完全接管；
- DAO长期没有独立存在的必要；
- DAO被确认是重复或空壳组织；
- 严重治理失效且无法通过休眠、重组或合并解决；
- 持续发生欺诈、利益输送或重大资金安全问题；
- 协议架构调整后该DAO不再需要。

永久撤销属于重要事项，不能通过普通日常乐观提案直接完成。

二级及更低级子DAO撤销流程为：
`提交撤销提案 → 直接上级DAO正式成员专业表决 → 进入全员复决窗口 → 决定生效 → 进入清算期`

一级DAO的撤销必须直接由全员投票决定。

撤销提案必须说明：

- 撤销原因；
- 是否已经尝试休眠、整改或合并；
- 未完成项目的处理方式；
- 已确认债务和劳动报酬的处理方式；
- 有争议资金的处理方式；
- 剩余资金的最终去向；
- 声誉和正式成员记录的处理方式；
- 合约权限和控制权的关闭方式；
- 清算期限；
- 历史档案保存方式。

**撤销后的清算**

撤销决定通过后，不应立即删除DAO或转移全部资金，应先进入清算期。

清算流程为：
`停止接收新增资金 → 停止创建新项目 → 完成、移交或终止已有项目 → 处理争议 → 支付债务和劳动报酬 → 处理剩余资金 → 撤销权限 → 永久归档`

清算期间：

- 不再接受新的普通任务；
- 不得建立新的长期项目；
- 已完成但尚未支付的劳动报酬优先支付；
- 已批准且仍有必要完成的项目可以继续；
- 无法继续的项目按照合同终止或移交；
- 有争议的资金继续锁定；
- 项目保证金和押金按照原规则处理；
- 清算人员不得修改已经通过的付款规则；
- 子DAO成员、创立者、提案人和清算执行者不得瓜分剩余资金。

**撤销后的剩余资金**

子DAO撤销后，剩余资金默认返回直接上级DAO本级资金池：
`子DAO清算后的剩余资金 → 直接上级DAO本级资金池`

撤销提案也可以明确将剩余资金：

- 转入承担其职责的继任DAO；
- 转入与其合并的DAO；
- 按明确比例转入多个承接DAO。

资金去向必须与撤销提案一起表决，不能在撤销通过后由清算执行者临时决定。

撤销后的资金迁移不属于长期闲置资金自动回收，而是经过治理正式批准的组织清算。

已经合法支付给贡献者的报酬不得追溯收回。

**撤销后的声誉和成员记录**

子DAO撤销后：

- 历史任务记录永久保留；
- 历史资金记录永久保留；
- 历史治理记录永久保留；
- 历史声誉记录永久保留；
- 已完成的真实劳动贡献不得删除；
- 该子DAO声誉不再产生该DAO的专业治理权；
- 已经计入上级DAO的声誉继续保留；
- 正式成员资格转为历史状态；
- 如果存在明确继任DAO，可以按照撤销提案中的规则映射相关专业声誉；
- 同一份声誉不得在继任DAO中重复生成。

撤销后的DAO应标记为：
`已解散`

不得从系统中完全删除，以便后续审计和查询。

**紧急暂停与永久撤销的区别**

发现资金漏洞、权限被盗或严重攻击时，任何人可以质押代币触发紧急安全暂停。

紧急暂停只能用于：

- 临时阻止资金继续损失；
- 暂停危险权限；
- 等待专业复核；
- 等待修复方案；
- 等待全员最终决定。

紧急暂停不能自动导致：

- 永久撤销子DAO；
- 永久没收子DAO资金；
- 删除成员和声誉；
- 把剩余资金直接转给上级DAO；
- 永久冻结所有权限。

永久撤销仍然必须经过正式治理程序和资产清算。

**决策权限**

不同类型子DAO事项的决策方式如下：


| 事项                      | 决策方式                                     |
| --------------------------- | ---------------------------------------------- |
| 建立试运行子DAO           | 乐观通过；有挑战时由上级DAO专业表决          |
| 正式成立二级及更低级子DAO | 直接上级DAO专业表决，并保留全员复决          |
| 成立一级DAO               | 直接全员投票                                 |
| 子DAO进入休眠             | 直接上级DAO专业表决，并保留全员复决          |
| 子DAO重新激活             | 乐观试运行或上级DAO专业表决                  |
| 同一上级下的子DAO合并     | 上级DAO专业表决，并保留全员复决              |
| 一级DAO合并               | 直接全员投票                                 |
| 二级及更低级子DAO撤销     | 上级DAO专业表决，并保留全员复决              |
| 一级DAO撤销               | 直接全员投票                                 |
| 紧急安全暂停              | 任何人质押后立即触发，随后专业复核和全员确认 |
| 永久资产清算              | 按已经通过的撤销或合并提案执行               |

所有二级及更低级子DAO的成立、休眠、合并和撤销决定，都可以在达到复决门槛后提交全员投票。

全员投票结果拥有最终效力。

## 资金池与资金分配

- 各级DAO资金池的新增资金来自协议每日增发的项目代币。
- 协议收入是已经发行的代币重新进入协议，应进入协议收入国库，与每日增发资金分别记账。
- 每枚具有治理资格的代币对应一份资金分配权。
- 持币者可以：
  - 自行分配各级资金池的票；
  - 把部分或全部资金分配权委托给专业人员；
  - 部分自行分配、部分委托；
  - 在同一领域委托给多名专业人员。
- 持币者可以随时修改或撤销自己的资金分配和委托配置。
- 修改从下一个每日结算周期开始生效，只影响未来新增代币。
- 已经进入某个DAO的资金不会因为持币者后来修改分配而移动。

例如，某持币者拥有 `1,000` 份有效资金分配票：

- 开发DAO：`600票`
- 推广DAO：`100票`
- 内容DAO：`300票`

开发DAO的 `600票` 可以继续配置为：

- 协议开发DAO：`200票`
- 前端DAO：`100票`
- 委托给开发专业人员A：`300票`

专业人员A只能在开发DAO及其下级DAO内分配这 `300票`，例如：

- 协议开发DAO：`150票`
- 基础设施DAO：`100票`
- 安全DAO：`50票`

最终该持币者在开发DAO子树中的分配为：

- 协议开发DAO：`350票`
- 前端DAO：`100票`
- 基础设施DAO：`100票`
- 安全DAO：`50票`

递归分配遵循以下规则：

- 同一份代币在整棵DAO树中只能计算一次。
- 分配给某个DAO的票，只能在该DAO及其下级DAO内继续分配。
- 开发DAO中的票不能转入推广DAO或内容DAO。
- 推广DAO中的票不能转入开发DAO或内容DAO。
- 内容DAO中的票不能转入开发DAO或推广DAO。
- 没有继续向下分配的票，对应新增资金保留在当前DAO本级资金池。
- 下级DAO获得的新增资金不能超过其直接上级DAO获得的新增资金。

某下级DAO获得的分配比例为：

`该下级DAO获得的有效分配票 ÷ 同一上级DAO范围内的全部有效票`

某下级DAO当日获得的新增资金为：

`直接上级DAO当日可分配资金 × 该下级DAO的有效分配比例`

例如开发DAO当日获得 `10,000` 枚新增代币，其内部有效分配为：

- 协议开发DAO：`40%`
- 前端DAO：`25%`
- 基础设施DAO：`20%`
- 安全DAO：`10%`
- 留在开发DAO本级：`5%`

则实际分配为：

- 协议开发DAO：`4,000枚`
- 前端DAO：`2,500枚`
- 基础设施DAO：`2,000枚`
- 安全DAO：`1,000枚`
- 开发DAO本级：`500枚`

保留在本级的资金可以用于：

- 跨下级DAO的联合项目；
- 尚未明确归类的任务；
- 本级公共支出；
- 应急支出；
- 为大型长期项目积累资金。
- 持币者转出代币后，其资金分配票按照有效余额自动减少。
- 持币者获得更多代币后，新增代币默认沿用该地址已有的分配和委托比例。
- 持币者也可以重新修改配置。
- DAO国库、协议收入国库、发行合约、项目托管合约等协议控制地址持有的代币不产生资金分配权。
- DAO不能使用自己持有的代币给自己分配更多新增资金。

基本原则：

`DAO拥有资金使用权，但DAO国库中的代币不具有资金分配权。`

**创世阶段的默认资金分配**

- 创世发行量为 `0`，协议启动时没有持币者，也没有个人资金分配票。
- 最初的每日新增代币按照协议预设的默认比例分配。
- 初始默认比例建议为：
  - 开发DAO：`70%`
  - 推广DAO：`10%`
  - 内容DAO：`20%`
- 初期较高的开发比例用于解决用户少、协议收入低，但开发支出较大的问题。
- 默认比例不是开发DAO的额外增发，只是统一每日增发资金的初始流向。
- 第一批新增代币进入各级DAO后，由DAO通过开发、推广和内容任务支付给贡献者。
- 创始人和初始团队没有预分配，也必须通过完成DAO任务获得代币。
- 尚未主动配置资金分配的代币按照默认比例计算。
- 持币者主动配置后，其个人配置覆盖默认比例。

**资金池已有余额**

- 已经进入某个DAO资金池的代币归该DAO管理。
- 不设置长期未使用资金的自动回收机制。
- 不因长期没有支出而自动退回上级DAO。
- 不因支持比例下降而自动转移已有余额。
- 不因DAO暂时没有项目而销毁余额。
- DAO可以长期积累资金，用于大型开发、安全审计、基础设施、推广活动或其他未来项目。
- 某个DAO资金过多时，持币者通过减少其未来分配票，降低之后的新增资金流入。
- 某个DAO被合并、撤销或永久停止运行时，已有余额按照预先制定的解散规则处理，不能由普通闲置规则自动处理。

每个DAO应公开：

- 当前资金池总余额；
- 已承诺但尚未支付的金额；
- 尚未承诺的可用余额；
- 最近 `7日`、`30日`和`90日`的新增资金流入；
- 最近 `7日`、`30日`和`90日`的实际支出；
- 当前获得的资金分配票数量和比例；
- 直接分配票和专业委托票的构成；
- 各专业人员获得的委托数量；
- 各专业人员的实际分配方案；
- 下级DAO的资金分配情况；
- 当前项目、预算、收款方、里程碑和付款记录；
- 提案人、执行者、验收者和收款方之间的关联关系。

# 任务与裁决

**任务的定义**

任务是具有明确目标、交付物、验收条件、奖励规则和责任主体，可以提交证据并经过挑战与裁决的一项贡献行为。

以下行为都可以按照相应规则视为完成任务：

- 评分；
- 打标签；
- 增加、合并、修改或废弃标签类别；
- 数据提供商批量入库小说；
- 数据提供商批量更新小说热数据；
- 用户手动入库一本小说；
- 完成邀请、渠道推广或社区运营任务；
- 开发代码；
- 修复程序问题；
- 完善文档；
- 部署和维护基础设施；
- 提交安全漏洞报告；
- 完成数据审核、任务验收和争议裁决；
- 各DAO认可的其他有效贡献。

评分、打标签、邀请新用户、手动登记小说等高频标准化操作，不需要逐项发布普通任务，可以由相关DAO预先制定统一任务规则，由任务模块自动创建任务记录、锁定奖励并完成结算。

代码开发、文档编写、批量数据服务、推广项目、基础设施维护等非标准化工作，应当创建独立任务。

基本原则：

`任务必须对应真实、必要、可验证的劳动或成果，不能仅以消耗时间、提交次数、账户数量或形式上的完成作为获得奖励和声誉的依据。`

**所有个人代币都通过任务模块发放**

发行合约不能直接向创始人、开发者、投资人、顾问或其他个人地址增发代币。

每日新增代币首先按照资金分配规则进入各级DAO资金池，然后只能通过任务模块进入个人钱包。

基本流转路径为：

`发行合约 → 各级DAO资金池 → 任务模块 → 任务执行者或其他收款人`

以下面向个人的代币发放都必须由任务模块执行：

- 开发任务劳动报酬；
- 数据提供奖励；
- 标签奖励；
- 评分奖励；
- 推广奖励；
- 审核奖励；
- 挑战奖励；
- 裁决劳动报酬；
- 漏洞报告奖励；
- 成本报销；
- 赔偿；
- 其他协议向个人支付的款项。

不同性质的付款必须分别记账。

任务总付款可以表示为：

$TaskPayment = LaborReward + Reimbursement + ProcurementCost + ReviewReward + BondRefund + OtherPayment$

其中：

- `LaborReward`：劳动报酬；
- `Reimbursement`：成本报销；
- `ProcurementCost`：采购费用；
- `ReviewReward`：审核奖励；
- `BondRefund`：保证金返还；
- `OtherPayment`：其他付款。

DAO不得绕过任务模块，以普通转账方式向个人支付劳动报酬或奖励。

DAO之间的资金调拨、资金池清算和协议内部账目调整，不属于面向个人的任务付款。

**标准劳动价值与声誉**

平台使用已经确认的人类劳动时间衡量任务声誉。

原则上：

`1分钟已经确认的人类劳动时间 = 1点声誉`

普通固定报酬任务的人类劳动报酬可以采用：

`1分钟人类劳动 = 1枚代币劳动报酬`

设任务需要的合理标准劳动时间为：

`HumanLaborMinutes`

则任务的标准声誉为：

$StandardReputation = HumanLaborMinutes$

人类劳动时间不是任务执行者自行申报的实际用时，而是根据以下因素预先评估或验收确认：

- 正常完成任务所需的合理时间；
- 任务难度；
- 专业要求；
- 交付物质量要求；
- 同类任务历史数据；
- 自动化和AI工具可以节省的工作量；
- 任务风险和责任；
- 任务能否重复使用；
- 任务对项目的实际价值。

使用AI或自动化工具提高效率，不自动减少执行者应获得的声誉。

但不得通过以下方式获得声誉：

- 虚报人工时间；
- 重复提交自动生成内容；
- 制造没有实际价值的工作；
- 人为拆分任务；
- 对同一成果重复创建任务；
- 通过关联账户互相验收虚假任务。

不同任务类型可以采用不同的代币报酬与声誉关系。代币报酬可以包含人类劳动报酬和其他开销补贴，声誉只根据已经确认的人类劳动时间生成。

普通固定报酬任务的人类劳动部分可以采用：

$LaborTokenReward = HumanLaborMinutes$

$ReputationReward = HumanLaborMinutes$

竞标型任务采用：

$TokenReward = WinningBid$

$ReputationReward = FixedReputation$

动态奖励任务采用预先公布的奖励和声誉公式。

追溯性奖励任务根据最终批准的标准劳动量确定声誉和代币报酬。

声誉表示经过任务规则确认的人类劳动贡献，不要求在所有情况下都等于执行者实际获得的代币数量。

**任务类型**

任务主要分为：

- 普通开放任务；
- 竞标型任务；
- 追溯性奖励任务；
- 动态奖励任务；
- 启动期普通任务；
- 紧急安全任务。

**普通开放任务**

普通开放任务由相关DAO预先确定任务要求，符合资格的用户可以领取。

创建任务时应当确定：

- 任务目标；
- 工作范围；
- 不包含的工作；
- 交付物；
- 验收条件；
- 固定劳动报酬；
- 固定任务声誉；
- 执行质押；
- 完成期限；
- 里程碑；
- 挑战期；
- 领取资格；
- 违约处理规则。

普通开放任务可以采用：

`先领取者获得执行权`

也可以要求：

`领取后在规定期限内提交开始证明`

领取者超过规定期限没有开始执行时：

- 任务领取资格失效；
- 根据造成的占用时间和影响，部分罚没执行质押；
- 任务重新开放。

普通固定报酬任务完全通过后：

$LaborTokenReward = HumanLaborMinutes$

$ReputationReward = FixedReputation$

通常可以设置：

$FixedReputation = HumanLaborMinutes$

**竞标型任务**

部分开发任务、数据提供商批量入库小说、数据提供商批量更新热数据、推广任务、基础设施维护任务等，可以采用竞标方式分配。

竞标型任务采用公开降价拍卖机制。

基本规则为：

- 任务创建时固定任务声誉；
- 任务创建时设置DAO愿意支付的最高代币报酬；
- 竞标者只对完成任务要求的代币报酬进行报价；
- 第一份报价不得超过最高代币报酬；
- 后续报价必须低于当前最低报价；
- 不允许提交等于或高于当前最低报价的报价；
- 竞标截止时，当前最低有效报价者自动中标；
- 中标价格决定实际代币报酬；
- 中标价格不改变任务创建时固定的声誉。

竞标者不需要提交：

- 自定义实施方案；
- 自定义声誉要求；
- 个人介绍；
- 工作经历说明；
- 其他由任务创建者主观比较的材料。

任务所需的能力、历史任务记录、DAO声誉、设备条件和服务能力，应当在任务创建时设置为客观最低要求，由任务模块自动验证。

**竞标型任务的创建**

竞标型任务创建时应当确定：

- 任务目标；
- 工作范围；
- 不包含的工作；
- 交付物；
- 验收条件；
- 完成期限；
- 固定任务声誉；
- 最高代币报酬；
- 最低允许报价；
- 最小降价幅度；
- 报价精度；
- 执行质押；
- 竞标开始时间；
- 竞标截止时间；
- 是否启用自动延时；
- 最低DAO声誉；
- 最低历史有效任务数量；
- 允许的最高违约率；
- 需要满足的技术、设备或服务能力；
- 同时执行任务数量上限；
- 违约罚没规则；
- 中标者违约后的处理方式；
- 其他可以由任务模块客观验证的资格条件。

竞标开始后，不得修改：

- 固定任务声誉；
- 最高代币报酬；
- 最低允许报价；
- 最小降价幅度；
- 验收条件；
- 竞标资格；
- 执行质押；
- 中标规则；
- 违约规则。

确实需要修改时，应当取消原任务，并按照任务取消规则处理后重新创建。

**竞标型任务的固定声誉**

竞标型任务的声誉在创建时直接指定，记为：

`FixedReputation`

固定任务声誉表示该任务正常完成时对应的标准劳动贡献，不随报价变化。

任务完全通过验收后：

$ReputationReward = FixedReputation$

执行者实际获得的代币报酬为：

$TokenReward = WinningBid$

例如：

- 固定任务声誉：`1,200点`；
- 最高代币报酬：`1,500枚`；
- 最终中标报价：`850枚`。

任务完成后，执行者获得：

- `850枚代币`；
- `1,200点`对应DAO声誉；
- 执行质押返还。

固定任务声誉不表示DAO必须支付相同数量的代币。

**竞标型任务的预算预留**

任务创建时，DAO应按照最高代币报酬预留资金。

需要预留的金额可以表示为：

$ReservedBudget = MaximumReward + MaximumReviewReward + ApprovedReimbursement$

其中：

- `MaximumReward`：最高代币报酬；
- `MaximumReviewReward`：最大审核奖励预算；
- `ApprovedReimbursement`：已经批准的成本报销。

竞标结束后，劳动报酬从最高代币报酬降为中标价格。

未被中标价格占用的预算重新成为DAO可用资金。

**竞标第一份报价**

尚未存在有效报价时，第一份报价必须满足：

$MinimumBid \le Bid_1 \le MaximumReward$

其中：

- `MinimumBid`：最低允许报价；
- `MaximumReward`：最高代币报酬；
- `Bid_1`：第一份有效报价。

第一名竞标者可以按照最高代币报酬报价，也可以直接提交更低价格。

**竞标后续报价**

已经存在有效报价后，新的报价必须严格低于当前最低报价。

设当前最低报价为：

`CurrentLowestBid`

则新报价必须满足：

$NewBid < CurrentLowestBid$

如果任务设置最小降价幅度：

`MinimumBidDecrement`

则新报价必须满足：

$NewBid \le CurrentLowestBid - MinimumBidDecrement$

同时必须满足：

$NewBid \ge MinimumBid$

例如：

- 当前最低报价：`1,000枚`；
- 最小降价幅度：`10枚`；
- 最低允许报价：`500枚`。

下一份有效报价必须满足：

$500 \le NewBid \le 990$

任务模块应当直接拒绝：

- 高于当前最低报价的报价；
- 等于当前最低报价的报价；
- 未达到最小降价幅度的报价；
- 低于最低允许报价的报价；
- 高于最高代币报酬的报价；
- 不符合报价精度的报价；
- 竞标资格不符合要求的报价；
- 未锁定执行质押的报价。

报价校验逻辑为：

    检查竞标是否仍在进行
    检查竞标者是否符合全部资格
    检查竞标者是否锁定足额执行质押
    检查报价是否符合代币精度
    检查报价是否不低于最低允许报价
    检查报价是否不高于最高代币报酬

    如果尚无有效报价：
        接受不高于最高代币报酬的有效报价
    否则：
        要求 NewBid <= CurrentLowestBid - MinimumBidDecrement
        不满足时直接拒绝

    接受报价
    更新 CurrentLowestBid
    更新 CurrentLowestBidder
    记录完整报价历史

因为每一份新报价都必须严格低于当前最低报价，所以不会出现两个相同的有效最低报价。

不需要设置：

- 同价时提交时间优先；
- 同价时随机抽签；
- 同价时二次竞价；
- 同价时人工选择。

**报价精度**

所有报价使用代币最小单位的整数表示，不使用浮点数。

例如代币精度为$10^{-6}$时：

- `1枚代币`记录为`1,000,000`个最小单位；
- `1.000001枚代币`记录为`1,000,001`个最小单位。

任务可以设置报价步长：

`BidStep`

有效报价必须满足：

$Bid \bmod BidStep = 0$

`BidStep`与`MinimumBidDecrement`可以设置为相同值，以简化竞标。

**竞标者再次报价**

同一竞标者可以多次降价。

竞标者的新报价仍然必须满足：

$NewBid \le CurrentLowestBid - MinimumBidDecrement$

竞标者不能：

- 提高自己的报价；
- 撤回已经公开的历史报价；
- 提交与当前最低价相同的报价；
- 提交高于当前最低价的报价；
- 将报价改回以前提交过的更高价格。

例如：

1. 用户A报价`1,000枚`；
2. 用户B报价`900枚`；
3. 用户A不能报价`950枚`；
4. 用户A只能提交低于`900枚`且满足最小降价幅度的报价。

全部历史报价永久保留。

只有当前最低报价具有中标资格。

**竞标资格**

竞标只有同时满足以下条件才有效：

- 竞标者符合任务规定的全部最低要求；
- 报价不高于最高代币报酬；
- 报价不低于最低允许报价；
- 报价达到最小降价幅度；
- 报价符合代币精度和报价步长；
- 竞标者已经锁定足额执行质押；
- 竞标尚未截止；
- 竞标者不存在禁止参与该任务的利益冲突；
- 竞标者没有处于任务资格暂停状态；
- 竞标者没有超过同时执行任务数量上限。

资格条件必须由任务模块自动验证。

任务创建者不能在竞标结束后增加新的资格要求或进行主观筛选。

**竞标执行质押**

竞标报价只决定实际支付的代币报酬，不决定执行质押。

执行质押由任务创建时统一确定。

可以采用：

$TaskBond = \max(MinimumTaskBond,\ MaximumReward \times TaskBondRate,\ ExpectedReplacementCost)$

其中：

- `MinimumTaskBond`：最低执行质押；
- `MaximumReward`：最高代币报酬；
- `TaskBondRate`：执行质押比例；
- `ExpectedReplacementCost`：中标者违约后重新发布任务和任务延误的预计成本。

所有竞标者原则上锁定相同数量的执行质押。

不得因为竞标者报价较低而降低其执行质押。

否则可能形成：

`报价越低 → 违约责任越低`

这会鼓励不负责任的恶意低价报价。

**竞标过程中的质押处理**

推荐规则为：

- 每个竞标者报价前锁定一份执行质押；
- 当前最低报价者的执行质押持续锁定；
- 被更低报价取代后，其执行质押立即解锁；
- 被取代者再次报价时，需要重新锁定执行质押；
- 当前最低报价者不能撤回报价；
- 当前最低报价者不能提前取回执行质押；
- 竞标截止后，中标者的执行质押继续锁定至任务最终结算。

采用该规则时，不自动保留第二低报价者作为候补执行者。

中标者违约后，任务重新开放竞标。

**竞标自动延时**

为了防止竞标者在截止前最后时刻提交最低报价，可以设置自动延时。

例如：

- 距离截止不足`10分钟`时出现新的最低报价；
- 竞标截止时间自动延长`10分钟`；
- 延长期间再次出现新的最低报价时继续延长；
- 最大累计延长时间为`60分钟`。

自动延时参数必须在竞标开始前确定。

只有新的有效最低报价可以触发延时。

无效报价不能触发延时。

**竞标中标规则**

竞标截止时，当前最低有效报价对应的竞标者自动中标：

$Winner = CurrentLowestBidder$

$WinningBid = CurrentLowestBid$

中标结果由任务模块自动确定。

任务创建者不能：

- 拒绝符合资格的最低报价者；
- 改选报价更高的竞标者；
- 根据竞标者身份改变结果；
- 根据额外提交的方案改变结果；
- 在竞标截止后增加资格要求；
- 在竞标结束后提高中标价格。

任务完全通过验收后：

$TokenReward = WinningBid$

$ReputationReward = FixedReputation$

并执行：

- 支付中标价格；
- 增加固定任务声誉；
- 退还执行质押；
- 释放未使用的预留预算；
- 记录最终交付物和验收结果。

**竞标最低允许报价**

任务可以设置：

`MinimumBid`

任何报价都必须满足：

$Bid \ge MinimumBid$

最低允许报价用于防止：

- 报价低于代币最小单位；
- 出现无法正常结算的极小报价；
- 通过异常零价格竞争固定声誉；
- 降价拍卖无限进行。

当前最低报价已经等于`MinimumBid`时，不再接受新的报价。

如果DAO允许无代币报酬完成任务，可以明确设置：

$MinimumBid = 0$

零代币中标者仍然必须：

- 锁定足额执行质押；
- 完成全部交付物；
- 通过完整验收；
- 经过正常挑战期。

只有任务最终有效，才能获得固定任务声誉。

**批量数据竞标任务**

数据提供商批量入库小说、批量更新热数据等任务，可以采用单位价格降价拍卖。

任务创建时确定：

- 预计数据数量；
- 最低有效数据数量；
- 最高单位代币报酬；
- 最低单位报价；
- 最小单位降价幅度；
- 单条有效数据对应的固定声誉；
- 数据字段和格式；
- 数据更新时间要求；
- 重复数据判断规则；
- 错误率上限；
- 抽样检查方式；
- 执行质押；
- 最终有效数量的计算方式。

竞标者提交每条最终有效数据所要求的代币报酬：

`UnitBid`

后续单位报价必须满足：

$NewUnitBid \le CurrentLowestUnitBid - MinimumUnitBidDecrement$

竞标截止时，单位报价最低者中标。

最终代币报酬为：

$FinalTokenReward = ValidRecordCount \times WinningUnitBid$

最终声誉为：

$FinalReputation = ValidRecordCount \times UnitReputation$

其中：

- `ValidRecordCount`：最终通过验收的有效记录数量；
- `WinningUnitBid`：中标单位报价；
- `UnitReputation`：单条有效数据对应的固定声誉。

以下记录不计入有效数量：

- 重复记录；
- 不满足入库条件的记录；
- 字段缺失记录；
- 格式错误记录；
- 数据来源无法验证的记录；
- 没有实际发生变化的热数据；
- 超过允许错误率后被判定无效的批次数据。

**追溯性奖励任务**

贡献者可以先完成具有公共价值的工作，再质押创建追溯性奖励任务，请求相关DAO支付报酬并生成声誉。

适用场景包括：

- 主动提交有价值的代码；
- 主动修复尚未发布任务的问题；
- 完善文档；
- 制作开发工具；
- 完成研究报告；
- 整理重要数据问题；
- 提交可以直接采用的改进方案。

追溯性奖励不是自动取得报酬的权利。

创建追溯性奖励任务时，应当提交：

- 完整成果；
- 成果首次公开时间；
- 代码、文档或数据链接；
- 解决的问题；
- 对项目的实际价值；
- 标准劳动时间估算；
- 请求的代币报酬；
- 请求的声誉；
- 是否已经从其他任务取得报酬；
- 使用的第三方成果和许可证；
- 与审核者、任务创建者的利益关系；
- 追溯性任务质押。

追溯性奖励任务应检查：

- 成果是否真实存在；
- 是否具有实际价值；
- 是否属于重复提交；
- 是否抄袭；
- 是否已经获得其他报酬；
- 请求金额是否合理；
- 请求声誉是否符合标准劳动量；
- 是否为了获取奖励制造不必要的工作；
- 是否应当归入其他已有任务。

追溯性奖励任务可以使用较高的任务创建质押和较长挑战期。

大额追溯性奖励任务可以要求至少一名无利益冲突的专业人员主动确认。

追溯性任务最终通过后：

- 支付最终批准的代币报酬；
- 生成最终批准的声誉；
- 退还任务创建质押。

**动态奖励任务**

评分、打标签、邀请新用户、手动登记小说、数据纠错等高频操作，可以由相关DAO预先制定动态任务规则。

动态任务规则应当提前公开：

- 奖励计算公式；
- 声誉计算公式；
- 单次奖励上限；
- 每日或每周期预算；
- 操作质押；
- 重复操作限制；
- 奖励资格；
- 挑战条件；
- 数据来源；
- 防女巫规则；
- 关联账户处理规则；
- 奖励追回方式。

动态任务的代币奖励可以表示为：

$DynamicTokenReward = f(ActionData,\ AvailableBudget,\ ProtocolParameters)$

动态任务的声誉可以表示为：

$DynamicReputation = g(ActionData,\ VerifiedContribution,\ ProtocolParameters)$

动态任务同样采用乐观确认：

`用户完成操作 → 计算并锁定奖励 → 进入挑战期 → 无有效挑战 → 自动结算`

动态奖励规则修改：

- 只影响生效后的新操作；
- 不追溯修改已经锁定的奖励；
- 应设置公开生效延迟；
- 不得在结果已经出现后临时修改结算方式。

动态任务必须设置周期预算上限，不能形成无限支付义务。

**启动期普通任务参数**

协议启动早期不需要单独设计只能执行一次的特殊启动任务。

启动期已经完成或正在完成的必要工作，包括协议部署、任务模块部署、初始DAO和资金池配置、初始前端发布、初始文档发布、测试网关服务器部署等，都应使用普通任务流程记录和发放报酬。

启动期普通任务可以使用临时宽松参数：

- 挑战期可以设置为几秒；
- 不要求任务创建质押；
- 不要求执行质押；
- 交付物必须能够通过链上记录、Process ID、交易ID、代码提交、代码哈希、部署记录或文档记录验证；
- 任务仍然必须记录所属DAO、执行者、交付物、收款钱包、人类劳动时间、代币报酬、开销补贴和声誉；
- 任务仍然通过任务模块付款；
- 任务仍然保留挑战、裁决、撤销声誉和剔除恶意所得的能力。

启动期任务不是预挖或预分配，因为：

- 协议部署完成时代币供应量仍然为`0`；
- 代币由统一发行合约或测试发行流程进入DAO资金池；
- 个人只能通过任务模块取得报酬；
- 任务交付物、付款、声誉和裁决记录全部公开；
- 后续可以通过治理延长挑战期、增加质押要求和调整任务资格。

**紧急安全任务**

发现可能导致资金损失、权限被盗、数据损坏或核心协议失效的安全问题时，可以创建紧急安全任务。

紧急安全任务可以包括：

- 漏洞报告；
- 紧急暂停；
- 漏洞复现；
- 修复代码；
- 安全审计；
- 资金追踪；
- 数据恢复；
- 事后分析。

紧急安全任务可以先提交保密证据。

修复完成前，不要求公开：

- 完整攻击路径；
- 关键代码位置；
- 私密密钥信息；
- 可以直接利用的复现细节。

紧急安全任务可以绕过普通任务发布等待期，但仍需：

- 提供最小必要证据；
- 锁定相应安全报告质押；
- 限制操作范围；
- 在事后进入完整审核和挑战流程。

漏洞报告奖励可以包括：

$SecurityReward = BaseLaborReward + RiskReward$

基础劳动奖励根据调查、复现和报告所需的标准劳动确定。

风险奖励根据实际保护的资金、权限和协议价值另行计算。

安全风险奖励不自动生成等额声誉。

声誉主要根据实际调查、报告和修复劳动确定。

**任务信息**

普通任务至少应记录：

- 任务ID；
- 所属DAO；
- 所属预算分类；
- 任务类型；
- 任务标题；
- 任务目标；
- 工作范围；
- 不包含的工作；
- 交付物；
- 验收条件；
- 代币报酬规则；
- 声誉规则；
- 成本报销；
- 采购费用；
- 审核奖励预算；
- 总预算；
- 任务创建质押；
- 执行质押；
- 挑战质押；
- 任务发布挑战期；
- 成果挑战期；
- 执行期限；
- 里程碑；
- 执行者；
- 收款钱包；
- 奖励钱包；
- 是否允许提前发放；
- 是否需要主动专业确认；
- 利益关系；
- 证据保存位置；
- 争议处理方式。

任务批准时必须锁定或预留足够预算。

不得在任务完成后，以资金不足为由拒绝支付已经批准并有效完成的任务。

**任务发布**

任何人可以按照DAO规则提交任务提案。

任务发布过程采用乐观通过：

`提交任务提案 → 进入任务发布挑战期 → 无有效挑战 → 任务生效`

任务发布阶段可以挑战：

- 任务没有实际价值；
- 预算明显不合理；
- 固定声誉明显过高；
- 验收条件无法执行；
- 与已有任务重复；
- 任务被人为拆分；
- 任务属于利益输送；
- 任务不属于该DAO职责；
- 所属预算不足；
- 关联关系未披露；
- 任务可能造成重大安全风险；
- 资格要求为特定人员量身定制；
- 竞标型任务的最低报价或最高报酬明显不合理。

任务发布通过，只表示DAO同意按照公布条件购买相应成果，不表示执行者提交任何内容都能获得付款。

任务发布者可以按照任务风险锁定任务创建质押。

任务提案被确认属于恶意、虚假、重复或利益输送时，可以罚没任务创建质押。

**任务质押**

除协议明确规定的低风险免费操作和启动期普通任务临时参数外，创建、领取、执行或挑战有奖励任务，原则上需要质押代币。

普通执行质押可以采用：

$TaskBond = \max(MinimumTaskBond,\ ReferenceReward \times TaskBondRate,\ ExpectedReviewCost)$

其中：

- `MinimumTaskBond`：最低任务质押；
- `ReferenceReward`：任务固定报酬或最高报酬；
- `TaskBondRate`：任务质押比例；
- `ExpectedReviewCost`：争议复核成本。

不同任务可以设置不同质押比例：

- 容易客观验证的任务可以降低质押；
- 主观性较强的任务可以提高质押；
- 大额任务可以采用里程碑质押；
- 追溯性奖励任务可以提高质押；
- 批量数据任务可以根据历史错误率调整质押；
- 有严重违约记录的执行者可以提高质押；
- 长期准确完成任务的执行者可以在规定范围内降低质押。

质押不能替代验收。

质押金额较高也不能使错误成果自动有效。

**任务状态**

任务可以具有以下状态：

- `Draft`：草稿；
- `ProposalChallenge`：任务发布挑战期；
- `Open`：等待领取；
- `Auction`：竞标中；
- `Claimed`：已经领取；
- `InProgress`：执行中；
- `Submitted`：已经提交成果；
- `EarlyPaid`：已经提前发放部分或全部款项；
- `Challenged`：成果受到挑战；
- `Finalized`：最终确认；
- `Rejected`：成果无效；
- `Cancelled`：任务取消；
- `Expired`：任务过期；
- `DebtOutstanding`：存在未偿任务债务。

**任务生命周期**

普通任务流程为：

`创建任务 → 发布挑战期 → 任务生效 → 领取或竞标 → 执行任务 → 提交成果 → 成果挑战期 → 最终确认 → 支付报酬并生成声誉`

追溯性任务流程为：

`提交已有成果 → 创建追溯性任务 → 挑战与审核 → 最终确认 → 支付报酬并生成声誉`

动态任务流程为：

`完成标准操作 → 自动创建任务记录 → 锁定奖励 → 挑战期 → 自动结算`

竞标型任务流程为：

`创建任务 → 发布挑战期 → 降价拍卖 → 最低报价者中标 → 执行任务 → 提交成果 → 挑战期 → 最终结算`

**任务成果提交**

执行者提交成果时，应提供足以完成验收的证据。

根据任务类型，可以包括：

- 代码提交记录；
- 代码哈希；
- 可复现测试结果；
- 部署记录；
- 文档链接；
- Arweave交易ID；
- 数据清单及哈希；
- 入库小说ID范围；
- 更新前后数据对比；
- 推广渠道记录；
- 费用凭证；
- 验收演示；
- 第三方测试结果；
- 其他可以验证任务完成情况的证据。

批量数据任务应当提交可验证的批次清单，至少记录：

- 批次ID；
- 数据提供商；
- 提交时间；
- 原始记录数量；
- 通过基本检查的数量；
- 拒绝记录数量；
- 重复记录数量；
- 最终有效记录数量；
- 抽样检查结果；
- 对应数据或交易哈希。

安全漏洞报告可以提交公开部分和保密部分。

**乐观验收**

任务成果原则上采用乐观确认：

`提交成果 → 进入成果挑战期 → 无有效挑战 → 自动确认`

无挑战不表示有人逐项审核，只表示在公开挑战期内没有出现足以阻止结算的有效异议。

挑战期可以根据任务风险设置：

- 容易客观验证的小额任务：较短挑战期；
- 普通开发、文档和数据任务：标准挑战期；
- 大额任务、关联方任务和复杂数据任务：较长挑战期；
- 核心协议、安全和重大不可逆任务：挑战期外增加主动专业确认。

任务最终确认后：

- 支付尚未支付的代币报酬；
- 支付符合条件的成本报销；
- 退还执行质押；
- 生成对应DAO声誉；
- 更新执行者任务记录。

**提前发放奖励**

为了避免执行者必须等待完整挑战期才能取得报酬，任务系统设置提前发放机制。

执行者提交成果时，可以选择是否启用提前发放：

- 默认启用；
- 执行者可以主动关闭；
- 关闭后按照普通挑战期结算；
- 提前发放不缩短挑战期；
- 提前发放不取消挑战权；
- 提前发放不代表任务已经最终确认。

其他用户审核成果后，可以质押代币，为提前发放提供审核担保。

审核担保表示：

`审核者认为该任务能够通过最终验收，并愿意以自己的质押承担判断错误的经济风险。`

审核者不是终局裁决者。

**提前发放的范围**

提前发放可以包括：

- 已提交成果对应的劳动报酬；
- 已完成里程碑对应的劳动报酬；
- 执行质押返还；
- 已经提供可靠凭证的成本报销。

未完成的后续里程碑不能提前发放。

本次提前释放的风险金额为：

$EarlyExposure = EarlyTokenReward + EarlyTaskBondRefund + EarlyReimbursement$

其中：

- `EarlyTokenReward`：提前支付的代币报酬；
- `EarlyTaskBondRefund`：提前退还的执行质押；
- `EarlyReimbursement`：提前支付的报销；
- `EarlyExposure`：协议在最终确认前承担的总风险。

**审核担保覆盖率**

所有有效审核者的质押总额记为：

`TotalReviewBond`

审核担保应达到规定覆盖率：

$TotalReviewBond \ge EarlyExposure \times ReviewCoverageRate$

其中：

- `ReviewCoverageRate`：审核担保覆盖率；
- 初始可以设置为`100%`。

未达到全部覆盖率时，可以只提前释放审核担保能够覆盖的金额。

提前释放上限为：

$EarlyReleaseLimit = {TotalReviewBond \over ReviewCoverageRate}$

实际提前释放金额为：

$ActualEarlyRelease = \min(RequestedEarlyRelease,\ EarlyReleaseLimit)$

审核担保必须持续锁定至任务最终确认。

审核者不能在挑战期中途撤回已经用于提前发放的质押。

达到提前释放所需的担保后，不再接受仅为了分取审核奖励的额外担保。

**审核者资格**

以下主体不得提供有效审核担保：

- 任务执行者本人；
- 任务收款钱包；
- 与执行者共同控制的账户；
- 与执行者存在未披露利益关系的人；
- 直接参与该任务并获得劳动报酬的人；
- 其他存在明显利益冲突的人。

任务可以根据风险要求审核者具备：

- 对应DAO正式成员资格；
- 最低累计声誉；
- 最低活跃声誉；
- 相关领域历史贡献；
- 最低审核准确率；
- 足够审核质押；
- 无严重错误审核记录。

普通小额任务可以允许任何用户通过足额质押提供审核担保。

核心协议、安全审计和重大不可逆任务，应当限制为具有相应专业资格且无利益冲突的审核者。

**审核奖励**

任务最终确认有效后，审核者获得审核奖励。

审核奖励比例初始设置为：

`ReviewRewardRate = 1%`

普通固定报酬任务的审核奖励为：

$ReviewReward = FinalTokenReward \times ReviewRewardRate$

竞标型任务的审核奖励按照实际中标报酬计算：

$ReviewReward = WinningBid \times ReviewRewardRate$

审核奖励应在任务预算中单独预留，不得从已经公布的执行者劳动报酬中扣除。

多名审核者共同提供担保时，审核奖励按照有效审核质押比例分配：

$ReviewerReward_i = ReviewReward \times {ReviewBond_i \over TotalReviewBond}$

其中：

- `ReviewBond_i`：审核者$i$的有效审核质押；
- `TotalReviewBond`：触发提前发放时的有效审核质押总额。

任务最终确认后：

- 审核质押全部退还；
- 审核奖励支付给审核者；
- 审核奖励不生成DAO声誉；
- 系统记录审核者的历史审核准确率。

审核奖励比例可以通过治理修改。

修改只影响生效后的新任务。

**提前发放后的挑战**

提前发放后，任务仍然继续原挑战流程：

`提前支付 → 挑战期继续 → 可以正常挑战 → 最终裁决`

如果任务最终完全有效：

- 执行者保留提前领取的款项；
- 审核者取回审核质押；
- 审核者获得审核奖励；
- 执行者获得任务声誉。

如果任务最终部分有效：

- 重新计算执行者应得代币报酬；
- 重新计算执行者应得声誉；
- 审核担保用于弥补提前支付超过最终应付金额的部分；
- 审核者按照担保比例承担损失；
- 剩余审核质押退还；
- 审核奖励按照最终有效代币报酬重新计算。

如果任务最终无效：

- 审核担保用于补回DAO已经提前支付的款项；
- 审核者按照各自有效担保比例承担损失；
- 审核者不获得审核奖励；
- 执行者不获得任务声誉；
- 已经生成的临时声誉撤销；
- 超出审核担保和其他可罚没质押的剩余损失，记为执行者任务债务。

如果存在审核者与执行者串通：

- 审核质押可以全部罚没；
- 取消审核奖励；
- 降低或取消审核资格；
- 提高后续审核质押要求；
- 情节严重时暂停任务系统权限。

审核者只是合理判断错误时，只承担已经承诺的审核担保损失。

除非存在明显草率、重复错误、利益关系隐瞒或串通证据，否则不追加声誉或资格处罚。

**任务挑战**

任何符合规则的人都可以质押代币挑战任务成果。

挑战必须指出具体问题，不能只提交笼统反对意见。

有效挑战理由包括：

- 任务没有完成；
- 交付物不符合验收条件；
- 数据错误率超过约定上限；
- 成果无法运行或无法复现；
- 成果属于抄袭；
- 同一成果重复领取奖励；
- 执行者隐瞒利益关系；
- 固定声誉与实际任务明显不匹配；
- 提交虚假凭证；
- 通过拆分任务重复获得报酬或声誉；
- 成果存在严重安全问题；
- 执行者和审核者存在未披露关联关系；
- 竞标任务没有按照最低报价规则执行；
- 其他明确违反任务规则的行为。

挑战质押可以采用：

$ChallengeBond = \max(MinimumChallengeBond,\ DisputedAmount \times ChallengeBondRate,\ ExpectedArbitrationCost)$

其中：

- `MinimumChallengeBond`：最低挑战质押；
- `DisputedAmount`：争议金额；
- `ChallengeBondRate`：挑战质押比例；
- `ExpectedArbitrationCost`：预计裁决成本。

任务受到挑战后：

- 暂停尚未发放的付款；
- 暂停生成正式声誉；
- 已提前发放的款项不立即撤回；
- 审核担保继续锁定；
- 执行者获得回应和补充证据的期限；
- 进入专业裁决流程。

**裁决程序**

普通任务争议首先由所属DAO处理。

流程为：

`任务受到挑战 → 挑战者提交证据 → 执行者回应 → 无利益冲突的专业人员裁决 → 进入复决窗口`

专业裁决采用对应DAO的专业治理规则：

`实际专业票权 = min（持币者有效专业委托票，活跃声誉 × 声誉承载系数）`

裁决者必须回避：

- 自己执行的任务；
- 自己创建并直接受益的任务；
- 自己提供审核担保的任务；
- 自己提出挑战的任务；
- 与自己存在利益关系的任务；
- 自己可能直接获得经济收益的任务。

重大任务争议、核心协议争议、大额资金争议或专业裁决存在重大分歧时，可以提交：

- 直接上级DAO；
- 安全DAO；
- 主DAO；
- 全体有效治理代币持有者复决。

全员复决结果拥有最终效力。

**裁决结果**

任务争议可以作出以下结果：

- 完全有效；
- 部分有效；
- 无效；
- 欺诈或恶意任务；
- 需要补充交付；
- 任务规则本身存在问题。

**完全有效**

任务被裁决为完全有效时：

- 执行者获得全部应得代币报酬；
- 执行质押全部退还；
- 生成全部任务声誉；
- 审核者取回审核质押并获得审核奖励；
- 挑战者质押根据规则部分罚没；
- 罚没部分用于支付裁决成本或进入相关DAO资金池。

**部分有效**

任务被裁决为部分有效时：

- 只确认已经实际完成并具有独立价值的部分；
- 按照预先规定的里程碑比例支付代币报酬；
- 按照预先规定的里程碑比例生成声誉；
- 根据违约程度部分罚没执行质押；
- 未使用预算返回相关DAO；
- 提前多支付部分由审核担保承担。

部分完成不能由裁决者在争议发生后任意估算。

任务允许部分完成时，必须在创建任务时预先设置可独立验收的里程碑。

设第$i$个里程碑的代币报酬权重为：

`TokenWeight_i`

声誉权重为：

`ReputationWeight_i`

应满足：

$\sum TokenWeight_i = 1$

$\sum ReputationWeight_i = 1$

最终代币报酬为：

$FinalTokenReward = TotalTokenReward \times \sum CompletedTokenWeight_i$

最终声誉为：

$FinalReputation = TotalReputation \times \sum CompletedReputationWeight_i$

任务不适合拆分时，应在创建时设置为：

`只能完全通过或完全失败`

**无效任务**

任务成果被裁决为无效时：

- 不支付尚未发放的代币报酬；
- 不生成任务声誉；
- 已经生成的临时声誉撤销；
- 部分或全部罚没执行质押；
- 审核担保用于补回提前支付损失；
- 剩余任务预算返回相关DAO；
- 挑战者获得合理挑战奖励。

**欺诈或恶意任务**

以下行为可以认定为欺诈或恶意任务：

- 伪造成果；
- 抄袭；
- 虚构费用；
- 对同一成果重复领取奖励；
- 与审核者串通；
- 隐瞒重大利益关系；
- 故意提交恶意代码；
- 提交虚假数据；
- 创建无实际价值的任务刷取声誉；
- 人为拆分任务增加声誉；
- 使用关联账户伪造竞标；
- 干扰最低报价拍卖；
- 其他明显骗取DAO资金或声誉的行为。

处理方式可以包括：

- 全额罚没任务创建质押；
- 全额罚没执行质押；
- 罚没审核担保；
- 撤销相关声誉；
- 追回尚可追回的奖励；
- 将未追回损失记为任务债务；
- 从未来奖励、质押返还或其他应付款中抵扣；
- 提高后续任务质押；
- 暂停领取任务资格；
- 暂停审核资格；
- 取消正式成员资格；
- 对关联参与者进行处罚。

**任务规则存在问题**

如果争议主要由任务本身描述不清、验收条件矛盾或任务创建者错误造成：

- 不应将全部责任推给执行者；
- 应当支付已经产生独立价值的有效劳动；
- 执行质押原则上退还；
- 任务创建质押可以用于补偿损失；
- 剩余任务可以修改后重新发布；
- 原任务记录永久保留。

**中标者违约**

竞标任务中标者存在以下行为时，可以认定为违约：

- 无正当理由拒绝执行；
- 中标后未在规定期限内开始；
- 主动放弃任务；
- 严重逾期；
- 交付物完全不符合验收条件；
- 提交虚假成果；
- 提交抄袭成果；
- 提交恶意代码或虚假数据；
- 隐瞒重大利益关系；
- 其他任务创建时明确规定的违约行为。

违约后：

- 取消中标资格；
- 不支付尚未支付的代币报酬；
- 不生成尚未生成的声誉；
- 撤销无效任务产生的临时声誉；
- 根据违约程度部分或全部罚没执行质押；
- 任务重新开放竞标。

原则上不自动让原第二低报价者中标，因为其执行质押已经解锁，执行条件也可能已经变化。

中标者已经完成部分有效里程碑时，按照预先规定的里程碑规则结算。

**里程碑任务**

中型和大型任务应拆分为可独立验收的里程碑。

每个里程碑应明确：

- 交付物；
- 代币报酬比例；
- 声誉比例；
- 截止时间；
- 验收条件；
- 对应执行质押；
- 是否允许提前发放；
- 与后续里程碑的依赖关系。

里程碑可以分别进入挑战期和结算。

已经最终确认的里程碑，不因后续里程碑失败自动撤销，除非：

- 前一里程碑本身存在造假；
- 前一里程碑无法独立使用，且任务创建时明确规定只有整体完成才有效；
- 执行者通过人为拆分骗取阶段付款。

**任务变更**

任务执行过程中需要修改范围、预算、期限或里程碑时，应提交任务变更提案。

变更提案应说明：

- 修改原因；
- 原任务内容；
- 新任务内容；
- 增减预算；
- 增减声誉；
- 已经完成的工作；
- 对执行者和DAO的影响；
- 是否存在利益关系。

不得在任务完成后修改验收条件，使原本不合格的成果自动合格。

不得在竞标结束后提高中标价格。

竞标型任务发生实质修改时，原则上应取消原任务并重新竞标。

**任务取消**

任务尚未被领取或尚未开始竞标时，可以按照规则取消，预留预算返回DAO。

任务已经被领取或已经中标后，DAO不能无理由单方面取消。

因DAO需求变化取消时：

- 验收已经完成的有效工作；
- 支付已经完成部分的合理代币报酬；
- 生成已经完成部分对应的声誉；
- 退还执行质押；
- 报销已经批准且无法撤销的合理成本。

因执行者违约取消时：

- 只支付已经独立产生价值并通过验收的部分；
- 只生成相应部分声誉；
- 根据违约程度罚没执行质押；
- 剩余预算返回DAO。

**重复成果与共同贡献**

同一成果不能通过多个任务重复获得代币报酬和声誉。

发现多个任务内容重复时，应判断：

- 哪个任务最先获得批准；
- 哪个成果最先公开提交；
- 后续成果是否具有独立增量价值；
- 多名贡献者是否真正独立完成；
- 是否存在抄袭或重复申领。

多人共同完成任务时，应在提交成果前明确：

- 各参与者的劳动分工；
- 代币报酬分配；
- 声誉分配；
- 收款钱包。

所有参与者分配的代币报酬之和不得超过任务最终代币报酬。

所有参与者分配的声誉之和不得超过任务最终声誉。

**利益冲突**

任务创建者、执行者、审核者、挑战者和裁决者应披露可能影响判断的利益关系。

需要披露的关系包括：

- 同一实际控制人；
- 家庭成员；
- 雇佣关系；
- 投资关系；
- 收益分成关系；
- 长期合作关系；
- 共同参与任务；
- 其他可能影响独立判断的关系。

关联方可以参与任务，但必须公开披露，并由无利益冲突的主体完成审核和裁决。

以下行为无效：

- 自己单独审核自己的任务；
- 自己单独裁决针对自己的挑战；
- 使用关联账户伪造独立审核；
- 隐瞒收益分成；
- 使用多个账户重复取得任务奖励；
- 通过关联账户制造虚假竞标；
- 创建者通过关联账户故意压低或操纵竞标价格。

**任务债务**

任务奖励已经提前支付，但后来被确认无效、重复、欺诈或应当减少时，无法立即追回的净损失记为任务债务。

净任务债务为：

$TaskDebt = InvalidEarlyPayment - SlashedReviewBond - SlashedTaskBond - RecoveredFunds$

任务债务可以从以下款项中抵扣：

- 未来任务奖励；
- 未来保证金返还；
- 未来审核奖励；
- 其他协议应付款；
- 治理允许抵扣的相关余额。

已经由审核者质押承担的损失，不再重复记为执行者向DAO偿还的债务，避免DAO获得重复补偿。

但执行者存在欺诈时，可以另行施加罚款和资格处罚。

任务债务记录绑定实际责任钱包，同时关联对应SNID的任务历史。

钱包迁移或SNID迁移不能消除已经产生的任务债务和处罚记录。

**任务审计记录**

任务系统应长期公开以下信息：

- 任务内容；
- 所属DAO；
- 所属预算分类；
- 固定声誉；
- 最高代币报酬；
- 竞标报价历史；
- 中标价格；
- 预算；
- 执行质押；
- 任务创建质押；
- 执行者；
- 交付物；
- 提交时间；
- 审核担保；
- 提前支付金额；
- 挑战记录；
- 裁决结果；
- 实际付款；
- 成本报销；
- 声誉生成和撤销；
- 利益关系；
- 任务债务；
- 相关链上交易；
- Arweave存证。

安全漏洞的技术细节可以在修复前保密。

但风险解除后，应公开：

- 任务金额；
- 暂停范围；
- 最终结论；
- 最终奖励；
- 质押处理结果；
- 最终修复状态。

**主要治理参数**

各DAO可以在上级规则允许的范围内设置：

- `MinimumTaskBond`：最低任务质押；
- `TaskBondRate`：任务质押比例；
- `MinimumChallengeBond`：最低挑战质押；
- `ChallengeBondRate`：挑战质押比例；
- `ProposalChallengePeriod`：任务发布挑战期；
- `CompletionChallengePeriod`：成果挑战期；
- `RetrospectiveBondRate`：追溯性任务质押比例；
- `ReviewCoverageRate`：提前发放审核担保覆盖率；
- `ReviewRewardRate`：审核奖励比例，初始为`1%`；
- `MaximumEarlyRelease`：单任务最大提前发放金额；
- `DynamicRewardBudget`：动态任务周期预算；
- `TaskDebtOffsetRate`：任务债务抵扣比例；
- `MinimumBidDecrement`：最小降价幅度；
- `BidStep`：报价步长；
- `AuctionExtensionWindow`：自动延时触发窗口；
- `AuctionExtensionDuration`：每次延时时间；
- `MaximumAuctionExtension`：最大累计延时时间；
- 各类任务的竞标、验收和抽样检查参数。

参数修改：

- 只影响生效后的新任务；
- 不追溯修改已经批准任务的报酬、声誉、竞标或质押条件；
- 应设置公开生效时间；
- 重大参数修改由对应DAO专业治理决定；
- 仍保留全员复决权。

**核心原则**

1. 所有面向个人的协议代币最终都通过任务模块发放。

2. 任务必须具有明确目标、交付物、预算、声誉规则和验收条件。

3. 任务原则上需要质押代币，执行者需要为成果质量承担经济责任。

4. 普通开放任务、竞标型任务、追溯性奖励任务和动态奖励任务使用不同的创建和定价方式。启动期普通任务只调整挑战期和质押参数，不作为独立任务类型。

5. 声誉反映经过确认的人类劳动时间，不用于补偿服务器、工具、采购等其他货币开销。

6. 代币也以人类劳动时间作为价值参考，但可以按照任务规则换算并补贴其他开销。

7. 竞标型任务在创建时固定声誉，实际代币报酬由公开降价拍卖决定。

8. 竞标型任务的后续报价必须低于当前最低报价。

9. 任务模块直接拒绝等于或高于当前最低报价的报价。

10. 竞标截止时，当前最低有效报价者自动中标。

11. 任务创建者不能在竞标结束后人工更换中标者。

12. 竞标价格只影响实际代币报酬，不影响固定任务声誉、验收条件和违约责任。

13. 低报价不能降低执行质押。

14. 所有任务发布和任务成果原则上采用乐观通过机制。

15. 高风险任务可以在挑战期外增加主动专业确认。

16. 提前发放只改变付款时间，不改变最终验收标准和挑战期限。

17. 提前发放由审核者质押担保，任务出错时审核者承担相应经济损失。

18. 审核奖励初始为最终有效代币报酬的`1%`，不生成声誉。

19. 同一成果不能重复获得任务奖励和声誉。

20. 任务执行者、审核者和裁决者必须披露利益关系。

21. 无效任务撤销奖励和声誉。

22. 欺诈任务还应承担质押罚没、任务债务和资格处罚。

23. 创世发行量保持为`0`，第一笔个人代币也必须来自任务模块中的普通任务付款。

24. 所有任务、付款、竞标、审核、挑战、裁决和债务记录都应当可以公开查询和审计。
